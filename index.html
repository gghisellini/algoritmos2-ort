1 ¿Qué es un AVL?

El nombre viene de sus inventores: Adelson-Velskii y Landis (1962). Es el primer árbol binario de búsqueda auto-balanceado que se inventó.

📐 Definición formal

Un AVL es un árbol binario de búsqueda (ABB) donde para cada nodo se cumple:

  • El subárbol izquierdo es un AVL
  • El subárbol derecho es un AVL
  • Se cumple el ordenamiento de ABB (menores a la izquierda, mayores a la derecha)
  • |altura(subárbol_izq) − altura(subárbol_der)| ≤ 1

Las tres propiedades clave que lo definen:

  • Binario: cada nodo tiene como máximo 2 hijos
  • De búsqueda: los elementos respetan una relación de orden
  • Balanceado: garantiza O(log n) para insertar, eliminar y buscar
💡 Clave para recordar

El caso base de la definición recursiva es: el árbol vacío es un AVL. Esto es fundamental para la recursión tanto en la definición como en la implementación.

Sección 1 — ¿Qué es un AVL?
AVL = Árbol Binario de Búsqueda auto-balanceado (Adelson-Velskii y Landis, 1962). Condición AVL: en cada nodo, |altura(izq) − altura(der)| ≤ 1. Es una definición recursiva. Caso base: el árbol vacío ES un AVL. 3 propiedades: Binario · De búsqueda (orden) · Balanceado (garantiza O(log n)).
Diferencia clave con ABB: el ABB no garantiza balance → puede tener altura O(n). El AVL siempre mantiene altura O(log n).

2 ¿Por qué conviene usar AVLs?

En un ABB común, todas las operaciones tienen complejidad O(h) donde h es la altura del árbol. El problema es que h puede ser O(n) si el árbol está sesgado (todos los nodos insertados en orden).

❌ ABB sesgado (peor caso)

1 \ 2 \ 3 \ 4 \ 5

Altura = n − 1 → operaciones O(n)

✅ AVL balanceado

3 / \ 2 4 / \ 1 5

Altura = O(log n) → operaciones O(log n)

OperaciónABB (peor caso)AVL (siempre)
BúsquedaO(n)O(log n)
InserciónO(n)O(log n)
EliminaciónO(n)O(log n)
Mínimo / MáximoO(n)O(log n)
🎯 La garantía del AVL

La altura de un AVL siempre es O(log n). Esto es lo que hace que TODAS las operaciones sean eficientes sin importar el orden de inserción.

Sección 2 — ¿Por qué usar AVL? PARCIAL
Problema del ABB: con inserción ordenada se vuelve una lista → altura O(n) → todo O(n). Ventaja del AVL: altura garantizada O(log n) independientemente del orden de inserción.
Complejidades AVL (siempre): Búsqueda: O(log n) Inserción: O(log n) Eliminación: O(log n) Mínimo/Máximo: O(log n) ABB en peor caso: todas esas operaciones son O(n).
📋 Patrón de respuesta en examen — "¿Por qué AVL sobre BST?"
Estructura tu respuesta en 3 partes: (1) el problema del BST (inserción ordenada → lista → O(n)), (2) cómo el AVL lo resuelve (rotaciones que mantienen |fb| ≤ 1), (3) el costo (O(log n) siempre garantizado, vs O(log n) promedio / O(n) peor caso en BST).

3 ¿Cuándo usar un AVL?

Usás un AVL cuando:

  • Los elementos tienen una relación de orden (se pueden comparar con < y >)
  • Necesitás eficiencia garantizada en búsqueda, inserción o eliminación
  • El problema exige operaciones en O(log n)
⚠️ ¿Por qué NO se usan siempre?
  • Mayor sobrecarga de memoria (almacenar alturas, punteros extra — aunque es discutible)
  • Complejidad de implementación — las rotaciones agregan complejidad cognitiva al código
  • Si los datos ya vienen razonablemente balanceados, un ABB simple puede ser suficiente
Sección 3 — ¿Cuándo usar AVL? PARCIAL
Usá AVL cuando necesitás eficiencia garantizada en búsqueda/inserción/eliminación y los elementos tienen relación de orden. Usá Hash cuando solo necesitás buscar/insertar/eliminar y el orden no importa — Hash da O(1).
AVL vs Hash — resumen en parcial: AVL: O(log n), soporta orden, mínimo/máximo, rangos Hash: O(1), NO soporta orden, NO mínimo/máximo AVL necesita que los elementos sean comparables (<, >) Hash solo necesita una función de hash para la clave Desventajas AVL: mayor memoria (almacena alturas), implementación más compleja que ABB.

4 Balance y cálculo de alturas

Factor de balance

Para cada nodo, definimos el factor de balance (FB) como:

FB(nodo) = altura(nodo->der) - altura(nodo->izq)
  • FB = 0 → perfectamente balanceado
  • FB = 1 → subárbol derecho 1 más alto (aceptable)
  • FB = -1 → subárbol izquierdo 1 más alto (aceptable)
  • FB > 1 → desbalance a la derecha
  • FB < -1 → desbalance a la izquierda

Función altura

La misma función que en un ABB, pero con un detalle crucial: debés manejar el caso NULL.

int altura(NodoAVL *nodo) {
    if (nodo == NULL)
        return -1;  // o 0, depende de la convención
    return nodo->altura;
}
💡 Convención de altura

Hay dos convenciones: árbol vacío = -1 (la hoja tiene altura 0) o árbol vacío = 0 (la hoja tiene altura 1). Sé consistente con la que elijas durante todo tu código y durante todo el parcial.

Encontrar el camino del desbalance

Cuando insertás un nodo, el desbalance solo puede ocurrir en el camino de inserción (los nodos por los que pasaste desde la raíz hasta la hoja nueva). Usamos la recursión al volver (no tail-recursive) para chequear el balance de cada nodo en ese camino.

Sección 4 — Factor de Balance y Alturas PARCIAL
Factor de Balance (FB): FB(nodo) = altura(nodo→der) − altura(nodo→izq) FB ∈ {-1, 0, 1} → nodo balanceado. FB < -1 o FB > 1 → desbalance → hay que rotar. Función altura: árbol vacío (NULL) = -1. Hoja = 0. int altura(NodoAVL *n) { if (n == NULL) return -1; return n->altura; } Actualizar altura: nodo→altura = max(altura(izq), altura(der)) + 1
El desbalance solo ocurre en el camino de inserción. Se detecta al volver de la recursión.

5 Rotaciones

Las rotaciones son el mecanismo que usa el AVL para recuperar el balance después de una inserción (o eliminación). Para cualquier desbalance causado por inserción, se necesita a lo sumo una rotación doble.

📐 Los 4 nodos clave: W, Z, Y, X
  • W = el nodo recién insertado
  • Z = el primer nodo desbalanceado que encontrás subiendo desde W hacia la raíz
  • Y = hijo de Z en el camino hacia W
  • X = nieto de Z en el camino hacia W

Ojo con X: X no siempre es W. X = W únicamente si W quedó exactamente como nieto de Z (a 2 niveles). Si W está más profundo, entonces X es algún ancestro de W que está en la posición de nieto de Z.

🔎 ¿Cuándo X = W y cuándo X ≠ W?

Ambos casos son posibles. Depende del bf de los nodos en el camino antes de insertar.

Caso A — X = W

Y es hoja (bf=0). W queda exactamente a 2 niveles de Z → X es W.

Antes — insertar W=0

3 (Z, bf=-1) / 1 (Y, hoja, bf=0)
→

Después

3 (Z, bf=-2 !) / 1 (Y, bf=-1) / 0 ← W = X

Z=3, Y=1, X=W=0

Caso B — X ≠ W

Y y X ya tienen hijos (bf=0 en ambos). W cae debajo de X.

Antes — insertar W=1

5 (Z, bf=-1) / \ 3(Y) 7 / \ 2(X) 4
→

Después

5 (Z, bf=-2 !) / \ 3(Y) 7 / \ 2(X) 4 / 1 ← W (X≠W)

Z=5, Y=3, X=2, W=1

❓ ¿No debería haber estado desbalanceado antes si W está tan profundo? No — y este es el punto clave. En el Caso B, verificá el árbol antes de insertar: Z tiene bf=-1 ✓, Y tiene bf=0 ✓, X tiene bf=0 ✓. El árbol es un AVL válido. Al insertar W, cada nodo en el camino con bf=0 absorbió la inserción actualizando su altura sin desbalancearse — pero ese aumento de altura se propagó hacia arriba hasta Z, que ya tenía bf=-1 y llegó a bf=-2.

Existen 4 tipos de rotaciones, que se agrupan en 2 simples y 2 compuestas:

Caso Izq-Izq Rotación a la derecha (horaria)

Y es hijo izquierdo de Z, X es hijo izquierdo de Y

Z / \ Y T4 / \ X T3 / \ T1 T2
→
Y / \ X Z / \ / \ T1 T2 T3 T4

Caso Der-Der Rotación a la izquierda (antihoraria)

Y es hijo derecho de Z, X es hijo derecho de Y

Z / \ T1 Y / \ T2 X / \ T3 T4
→
Y / \ Z X / \ / \ T1 T2 T3 T4

Caso Izq-Der Rotación Izquierda en Y, luego Derecha en Z

Y es hijo izquierdo de Z, X es hijo derecho de Y

Z / \ Y T4 / \ T1 X / \ T2 T3
→
(Rot izq en Y) Z / \ X T4 / \ Y T3 / \ T1 T2
→
(Rot der en Z) X / \ Y Z / \ / \ T1 T2 T3 T4

Caso Der-Izq Rotación Derecha en Y, luego Izquierda en Z

Y es hijo derecho de Z, X es hijo izquierdo de Y

Z / \ T1 Y / \ X T4 / \ T2 T3
→
(Rot der en Y) Z / \ T1 X / \ T2 Y / \ T3 T4
→
(Rot izq en Z) X / \ Z Y / \ / \ T1 T2 T3 T4
Sección 5 — Las 4 Rotaciones PARCIAL
W = nodo recién insertado. Z = primer nodo desbalanceado (subiendo desde W). Y = hijo de Z en el camino hacia W. X = nieto de Z en el camino hacia W. ¿X = W? Solo si W quedó exactamente a 2 niveles de Z. Si W está más profundo, X es un ancestro de W (no W mismo).
Izq-Izq (Y hijo izq de Z, X hijo izq de Y) → Rotación DERECHA en Z (simple) Der-Der (Y hijo der de Z, X hijo der de Y) → Rotación IZQUIERDA en Z (simple) Izq-Der (Y hijo izq de Z, X hijo der de Y) → Rot IZQUIERDA en Y, luego DERECHA en Z (doble) Der-Izq (Y hijo der de Z, X hijo izq de Y) → Rot DERECHA en Y, luego IZQUIERDA en Z (doble)
Regla mnemónica: camino recto (Izq-Izq / Der-Der) → rotación simple en dirección opuesta. Camino en zigzag → rotación doble (primero enderezás, luego rotás). Después de rotar: siempre actualizar alturas — primero el nodo que bajó (Z), luego el que subió (Y o X).
📋 Patrón de respuesta en examen — "Describí las rotaciones dobles"
(1) Identificá el caso: el camino Z→Y→X hace zigzag (Izq-Der o Der-Izq). (2) Explicá la doble rotación: primero rotás en Y para convertirlo en caso recto, luego rotás en Z. (3) Indicá el orden: las rotaciones son O(1) — no dependen de n, son solo reasignaciones de punteros.

7 Implementación en C/C++

Estructura del nodo

struct NodoAVL {
    int dato;
    NodoAVL *izq;
    NodoAVL *der;
    int altura;
};

Funciones auxiliares

int altura(NodoAVL *n) {
    if (n == NULL) return -1;
    return n->altura;
}

int max(int a, int b) {
    return (a > b) ? a : b;
}

int balance(NodoAVL *n) {
    if (n == NULL) return 0;
    return altura(n->der) - altura(n->izq);
}

Rotación a la derecha (horaria)

NodoAVL* rotarDerecha(NodoAVL *z) {
    NodoAVL *y = z->izq;
    NodoAVL *T3 = y->der;

    // Realizar rotación
    y->der = z;
    z->izq = T3;

    // Actualizar alturas (primero z, luego y)
    z->altura = max(altura(z->izq), altura(z->der)) + 1;
    y->altura = max(altura(y->izq), altura(y->der)) + 1;

    return y; // nueva raíz
}

Rotación a la izquierda (antihoraria)

NodoAVL* rotarIzquierda(NodoAVL *z) {
    NodoAVL *y = z->der;
    NodoAVL *T2 = y->izq;

    // Realizar rotación
    y->izq = z;
    z->der = T2;

    // Actualizar alturas
    z->altura = max(altura(z->izq), altura(z->der)) + 1;
    y->altura = max(altura(y->izq), altura(y->der)) + 1;

    return y;
}

Inserción completa

NodoAVL* insertar(NodoAVL *nodo, int dato) {
    // 1. Inserción normal de ABB
    if (nodo == NULL)
        return crearNodo(dato);

    if (dato < nodo->dato)
        nodo->izq = insertar(nodo->izq, dato);
    else if (dato > nodo->dato)
        nodo->der = insertar(nodo->der, dato);
    else
        return nodo; // duplicados no permitidos

    // 2. Actualizar altura de este nodo
    nodo->altura = max(altura(nodo->izq), altura(nodo->der)) + 1;

    // 3. Calcular factor de balance
    int fb = balance(nodo);

    // 4. Si hay desbalance, aplicar rotación

    // Caso Izq-Izq
    if (fb < -1 && dato < nodo->izq->dato)
        return rotarDerecha(nodo);

    // Caso Der-Der
    if (fb > 1 && dato > nodo->der->dato)
        return rotarIzquierda(nodo);

    // Caso Izq-Der
    if (fb < -1 && dato > nodo->izq->dato) {
        nodo->izq = rotarIzquierda(nodo->izq);
        return rotarDerecha(nodo);
    }

    // Caso Der-Izq
    if (fb > 1 && dato < nodo->der->dato) {
        nodo->der = rotarDerecha(nodo->der);
        return rotarIzquierda(nodo);
    }

    return nodo; // sin desbalance
}
⚠️ Errores comunes en parciales
  • Olvidar actualizar la altura antes de chequear el balance
  • Actualizar las alturas en el orden incorrecto dentro de la rotación (primero Z, luego Y)
  • No manejar el caso NULL en la función de altura
  • Invertir el sentido de la resta del factor de balance y no ser consistente
  • Olvidar que la inserción debe retornar un nodo porque la raíz puede cambiar

Estructura del nodo AVL: dato, *izq, *der, altura (campo extra vs ABB). Rotación derecha (caso Izq-Izq): NodoAVL* rotarDerecha(NodoAVL *z) { NodoAVL *y = z->izq; NodoAVL *T3 = y->der; y->der = z; z->izq = T3; z->altura = max(altura(z->izq), altura(z->der)) + 1; // z primero y->altura = max(altura(y->izq), altura(y->der)) + 1; // y después return y; // nueva raíz } Inserción — 4 pasos al volver de la recursión: 1. nodo->altura = max(alt_izq, alt_der) + 1 2. int fb = altura(der) - altura(izq) 3. if |fb| > 1 → identificar caso Z/Y/X 4. aplicar rotación y retornar nueva raíz
⚠️ La inserción NO es tail-recursive — hace trabajo al volver. Debe retornar NodoAVL* porque la raíz puede cambiar.

8 Algoritmo paso a paso

Cada vez que insertás un nodo, este es el proceso completo:

Insertar el nodo — Usás la misma lógica de inserción de un ABB (bajar por izquierda o derecha según la comparación)

Calcular la altura — Al volver de la recursión, actualizás la altura de cada nodo en el camino: max(alt_izq, alt_der) + 1

Verificar el balance — Calculás FB = alt_der - alt_izq. Si |FB| > 1, hay desbalance

Identificar el caso — Determinás si es Izq-Izq, Der-Der, Izq-Der o Der-Izq mirando la relación entre Z, Y y X

Aplicar la rotación — Ejecutás la rotación correspondiente (simple o doble)

Recalcular alturas — Dentro de la rotación, actualizás las alturas de los nodos involucrados (primero el que baja, luego el que sube)

🔑 Concepto clave: recursión no tail-recursive

La inserción en un ABB común puede ser tail-recursive (no hace nada después de la llamada recursiva). En un AVL, necesitás ejecutar tareas al volver de la recursión (actualizar alturas y chequear balance), por lo que la función NO es tail-recursive. Esto es lo que permite detectar desbalances en el camino de vuelta.

Sección 8 — Algoritmo completo de inserción PARCIAL
Proceso de inserción AVL — 6 pasos: 1. Bajar por el árbol como ABB normal (izq si menor, der si mayor) 2. Al llegar a NULL → crear el nodo nuevo (hoja) 3. AL VOLVER de la recursión: actualizar altura del nodo actual 4. AL VOLVER: calcular FB = altura(der) − altura(izq) 5. Si |FB| > 1: identificar caso (Z/Y/X) y aplicar rotación 6. Retornar el nodo (puede ser la nueva raíz post-rotación)
Recursión NO tail-recursive = hace trabajo al volver (pasos 3-6). El ABB SÍ es tail-recursive (no hace nada al volver). Esta diferencia sale en parciales. Desbalance solo en el camino de inserción: solo los nodos por los que bajaste pueden quedar desbalanceados. El primer desbalanceado que encontrás al subir es Z.

Ejercicio 5 Medio
Explicá con tus palabras: ¿por qué la inserción en un AVL no puede ser tail-recursive? ¿Qué operaciones se ejecutan "a la vuelta" de la recursión?
Respuesta

En un ABB simple, la inserción puede ser tail-recursive porque después de insertar no hay más trabajo. En un AVL, al volver de cada llamada recursiva hay que: (1) actualizar la altura del nodo actual, (2) calcular el factor de balance, (3) si hay desbalance, aplicar la rotación correspondiente. Estas operaciones post-recursivas impiden que sea tail-recursive.

1 Tabla vs. Tabla de Hash

Primero hay que distinguir dos cosas que suenan parecidas pero son diferentes:

📋 Tabla (TAD)

Tipo abstracto de datos. Define qué puede hacer, no cómo. Asocia una clave a un valor. Sus operaciones son:

  • Búsqueda: dada una clave → retorna el valor
  • Insertar: dada clave + valor → inserta
  • Eliminar: dada una clave → elimina el valor

📗 Las claves deben ser únicas.

#️⃣ Tabla de Hash (Estructura)

Una implementación del TAD Tabla. Define cómo lo hace. Su superpoder:

  • Búsqueda: O(1)
  • Insertar: O(1)
  • Eliminar: O(1)

📘 Los elementos NO necesitan un orden definido. A diferencia de ABB/AVL.

🎯 La gran diferencia con AVL

En un AVL los elementos deben tener una relación de orden (comparables con < y >). En una tabla de hash no importa el orden — solo necesitás una función de hash para la clave. La tabla de hash te da O(1), el AVL te da O(log n). Pero el AVL te permite listar ordenado, buscar el mínimo/máximo, etc. La tabla de hash no.

Sección 1 — Tabla vs. Tabla de Hash PARCIAL
Tabla (TAD): tipo abstracto. Define qué hace: buscar(clave), insertar(clave, valor), eliminar(clave). No dice cómo. Tabla de Hash: implementación concreta del TAD Tabla. Garantiza O(1) promedio en las 3 operaciones.
Hash vs AVL — comparación de parcial: Hash: O(1) búsqueda/inserción. NO requiere orden. NO soporta mínimo/máximo. AVL: O(log n) todo. SÍ requiere orden. SÍ soporta mínimo/máximo, rangos. Usar Hash cuando: solo querés buscar/insertar/eliminar rápido. Usar AVL cuando: necesitás orden, rangos, recorrido ordenado.

3 La función de hash

La función de hash toma la clave y la convierte en un número entero que se usa como índice del array.

📐 Índice final

El índice nunca puede ser mayor que el tamaño del array, así que siempre hacemos:

index = hash(key) % cantidadDeBuckets

Características obligatorias

  • Convierte la clave a un entero — puede tomar un string, un objeto, lo que sea
  • Determinista — para la misma clave, siempre retorna el mismo valor. Sin aleatoriedad.

Características de una buena función de hash ✅

  • Evitar colisiones 💢 — distribuir los elementos lo más uniformemente posible entre los buckets
  • O(1) en tiempo — calcularla debe ser rápido, si no pierde sentido toda la estructura
  • Efecto avalancha 🏔️ — un pequeño cambio en la clave produce un resultado completamente distinto

Las 3 funciones del profesor — comparadas

// Hash 1: suma simple de caracteres (débil — "abc" == "bca" == "cab")
int hash1(string key) {
    int h = 0;
    for (int i = 0; i < key.length(); i++)
        h = h + key[i];
    return h;
}

// Hash 2: suma ponderada por posición (mejor — diferencia orden)
int hash2(string key) {
    int h = 0;
    for (int i = 0; i < key.length(); i++)
        h = h + key[i] * (i + 1);
    return h;
}

// Hash 3: multiplicación por primo (buena distribución — efecto avalancha)
int hash3(string key) {
    int h = 0;
    for (int i = 0; i < key.length(); i++)
        h = 31 * h + int(key[i]);
    return h;
}
⚠️ El problema de hash1

hash1("abc") == hash1("bca") == hash1("cab") — todas dan el mismo resultado porque solo suma. hash3 con el primo 31 es la base de la función hash de Java para strings (String.hashCode()) porque produce mejor distribución y efecto avalancha.

💡 ¿Existe la función de hash perfecta?

Sí, pero solo cuando conocés de antemano todos los elementos que vas a guardar (se llama perfect hashing). En la práctica general, no existe. Siempre hay posibilidad de colisión.

Sección 3 — La función de hash PARCIAL
Obligatorio: determinista (misma clave → siempre mismo resultado, sin aleatoriedad). 4 propiedades de una BUENA función de hash: 1. Determinista: misma entrada → siempre misma salida 2. Distribución uniforme: minimiza colisiones, usa todos los buckets 3. O(1): calcularla debe ser rápido (si no pierde sentido) 4. Efecto avalancha: pequeño cambio en clave → resultado totalmente distinto
hash1 (suma simple): débil — hash("abc") == hash("bca") == hash("cab"). No hay efecto avalancha. hash3 (×primo 31): mejor distribución + efecto avalancha. Base del String.hashCode() de Java. hash3: h = 31 * h + int(key[i]) // para cada carácter Índice final siempre: index = hash(key) % cantidadDeBuckets

4 Colisiones 💢

Una colisión ocurre cuando dos claves distintas producen el mismo índice:

hash("abc") % 10 == 5
hash("xyz") % 10 == 5   ← ¡colisión!

¿De qué depende la cantidad de colisiones?

  • Calidad de la función de hash — a mayor distribución, menos colisiones
  • Factor de carga (λ) — tabla más llena = más colisiones
🎯 ¿Por qué las colisiones son tan malas?

Cuando hay colisión, las operaciones dejan de ser O(1). En el peor caso, si todo colisiona en el mismo bucket, buscar es O(n). La tabla de hash pierde todo su atractivo.

Ejemplo conceptual: la paradoja del cumpleaños — con solo 23 personas en un cuarto, la probabilidad de que dos compartan cumpleaños es >50%. Con funciones hash pasa algo similar: los buckets se "llenan" antes de lo esperado.

Sección 4 — Colisiones y estrategias PARCIAL
Colisión: hash("abc") % 10 == hash("xyz") % 10 → dos claves distintas → mismo bucket. ¿De qué depende? Calidad de la función de hash + factor de carga λ. A mayor λ → más colisiones. Impacto: con muchas colisiones las operaciones se degradan a O(n) en el peor caso.
Estrategias de resolución — 4 que hay que saber nombrar: HASH ABIERTO (separate chaining): · Cada bucket tiene una lista enlazada · λ puede ser > 1 HASH CERRADO (open addressing) — 3 variantes: · Probing Lineal: index = (h(k) + i) % B · Probing Cuadrático: index = (h(k) + i²) % B · Double Hashing: index = (h(k) + i·h2(k)) % B

5 Hash Abierto (Separate Chaining)

Cada bucket contiene una lista enlazada. Cuando hay colisión, el nuevo elemento se agrega a la lista del bucket correspondiente.

Bucket 0: [ ] → null Bucket 1: [ ] → ("banana", 2) → null Bucket 2: [ ] → ("apple", 5) → ("grape", 1) → null ← colisión, ambas en bucket 2 Bucket 3: [ ] → null Bucket 4: [ ] → ("mango", 3) → null

Ventajas del hash abierto

  • Fácil de implementar
  • El λ puede ser mayor a 1 (más elementos que buckets) — nunca está "lleno"
  • Menos sensible a la calidad de la función de hash
  • Eliminación simple — solo removés de la lista
💡 Complejidad con hash abierto

En promedio O(1 + λ). Si λ es bajo (ej: 0.75), es prácticamente O(1). Si λ es alto (ej: 5), cada búsqueda recorre en promedio 5 nodos → se degrada.

Sección 5 — Hash Abierto (Separate Chaining) PARCIAL
Cada bucket = lista enlazada. Colisión → agregar al final (o inicio) de la lista del bucket. λ PUEDE ser > 1 — si hay 100 elementos en 10 buckets, λ=10 y cada lista tiene en promedio 10 elementos. Complejidad: O(1 + λ) promedio. Si λ ≤ 1, prácticamente O(1). Eliminación: simple — removés el nodo de la lista enlazada. No necesita tombstones.
Impacto de λ alto en hash abierto: listas largas → más comparaciones → se degrada a O(n). El tamaño de tabla no limita cuántos elementos podés insertar. Ventaja clave vs cerrado: nunca está "lleno", eliminación sencilla, menos sensible a la calidad del hash.

6 Hash Cerrado (Open Addressing)

Todos los elementos viven dentro del array. Cuando hay colisión, se busca el siguiente bucket disponible mediante una técnica de probing.

⚠️ Restricciones del hash cerrado
  • λ nunca puede ser mayor a 1 — si la tabla está llena, no podés insertar
  • La eliminación es más complicada (no podés simplemente borrar — rompe las cadenas de probing)
  • Debés detectar loops infinitos en el probing
  • Se aconseja usar números primos para el tamaño de la tabla

Los 3 tipos de probing

Lineal Probing Lineal

index(key, i) = (h(key) + i) % B

Prueba el siguiente bucket, luego el siguiente, etc. Produce clustering primario — los elementos se agrupan en zonas, empeorando con el tiempo.

Cuadrático Probing Cuadrático

index(key, i) = (h(key) + i²) % B

Salta más lejos con cada intento. Evita clustering primario pero produce clustering secundario — misma secuencia para claves que mapean al mismo bucket.

Doble Hash Double Hashing

index(key, i) = (h(key) + i * h2(key)) % B

Usa una segunda función de hash para el salto. La mejor distribución de los tres. Evita ambos tipos de clustering.

int calculateIndex(K key, unsigned int tryCount) {
    if (type == LINEAR_PROBING)
        return abs(hashFunction(key) + tryCount) % buckets;
    else if (type == QUADRATIC_PROBING)
        return abs(hashFunction(key) + pow(tryCount, 2)) % buckets;
    else if (type == DOUBLE_HASHING)
        return abs(hashFunction(key) + tryCount * hashFunction2(key)) % buckets;
}
// Nota: tryCount (i) es el número de intento (0, 1, 2, ...)

7 Factor de carga y Rehashing

📐 Factor de carga λ
λ = N / B
N = cantidad de elementos insertados
B = cantidad de buckets (tamaño del array)

La fórmula es la misma para hash abierto y cerrado.

λ bajo (ej: 0.3)

Pocas colisiones. Operaciones muy cercanas a O(1). Pero desperdiciás mucha memoria — muchos buckets vacíos.

λ alto (ej: 0.9+)

Muchas colisiones. Las operaciones se degradan. En hash cerrado no podés superar λ=1.

Rehashing ♻️

Cuando λ supera un umbral (comúnmente 0.75 en práctica), se hace un rehash:

Crear un nuevo array — generalmente el doble del tamaño anterior (preferiblemente número primo)

Recalcular todos los índices — cada elemento se re-hashea con el nuevo tamaño de tabla

Reinsertar todos los elementos — en sus nuevas posiciones

⚠️ El rehash es muy costoso

Tiene complejidad O(N) promedio — hay que tocar cada elemento. Por eso debe planificarse cuidadosamente. En Java, HashMap hace rehash al llegar a λ=0.75 y duplica el tamaño.

¿Cuándo usar tablas de hash?

✅ Son excelentes para:

  • Buscar, insertar y eliminar rápido
  • Conteo de frecuencias
  • Implementar diccionarios/mapas
  • Caché / memoización

❌ No sirven para:

  • Listar elementos en orden
  • Buscar el mínimo o máximo
  • Operaciones que dependen de relaciones entre elementos

Con el doble puntero, NULL es una señal limpia y universal de "esta celda no tiene nada".
② bool *eliminado — El patrón Tombstone (lápida)
Cuando eliminás un elemento en hash cerrado, NO podés simplemente poner tabla[i] = NULL. Si lo hacés, rompés las cadenas de probing: una búsqueda posterior podría encontrar ese NULL y asumir que el elemento buscado no existe, cuando en realidad está más adelante en la cadena.

La solución: marcar la celda como "eliminada" con eliminado[i] = true. Así el probing sabe que debe seguir buscando aunque la celda esté vacía. Este patrón se llama tombstone (lápida).
③ int N — Cantidad de elementos
Contador de elementos activos en la tabla. Se usa para calcular el factor de carga λ = N/B.
④ int B — Cantidad de buckets
El tamaño del array. En hash cerrado, B debe ser mayor que N (no podés insertar más elementos que buckets). Idealmente B es número primo — reduce colisiones al hacer la distribución más uniforme.
⑤ int (*hashFunction)(K) — Puntero a función
En lugar de hardcodear una función de hash, el profesor pasa la función como parámetro al constructor. La sintaxis int (*nombre)(K) significa "puntero a función que recibe un K y devuelve int". Esto hace la implementación flexible: podés probar distintas funciones de hash sin cambiar el código de la tabla.

¿Por qué NO podés hacer simplemente tabla[p] = NULL?

Si eliminás A (tabla[3]=NULL), una búsqueda de C vería NULL en pos 3 y devolvería false, aunque C existe en pos 7. Romperías la cadena.
La solución correcta — patrón Tombstone: 
void remove(K key) {
    assert(this->exists(key));      // precondición
    for (int i = 0; i < B; i++) {
        int p = pos(key, i);
        // Si encontramos celda vacía no-tombstone: no puede estar
        if (tabla[p] == NULL && !eliminado[p]) return;
        // Si encontramos el elemento:
        if (tabla[p] != NULL && !eliminado[p] && tabla[p]->clave == key) {
            delete tabla[p];         // liberar memoria
            tabla[p] = NULL;         // limpiar el puntero
            eliminado[p] = true;     // marcar como tombstone
            N--;                     // decrementar contador
            return;
        }
    }
}
¿Qué hace el tombstone exactamente?
Al poner eliminado[p] = true con tabla[p] = NULL:
  • Las búsquedas que pasan por esa posición no cortan la cadena (porque es tombstone, no vacío real)
  • Las inserciones nuevas pueden reusar esa posición (si llegaran a ella por probing)
  • La memoria del ClaveValor se libera con delete

¿Qué debe hacer rehash?
  1. Guardar referencia a la tabla y eliminado viejos
  2. Crear una tabla nueva del doble de tamaño (idealmente próximo número primo)
  3. Reinsertar solo los elementos activos (que no sean NULL ni tombstone)
  4. Liberar la memoria de la tabla vieja
Implementación guiada: 
void rehash() {
    // Guardar estado viejo
    ClaveValor<K,V>** tablaVieja = tabla;
    bool* eliminadoViejo = eliminado;
    int B_viejo = B;

    // Nuevo tamaño: doble (simplificado; idealmente número primo)
    B = B * 2;
    N = 0;

    // Crear nueva tabla limpia
    tabla = new ClaveValor<K,V>*[B]();
    for (int i = 0; i < B; i++) tabla[i] = NULL;
    eliminado = new bool[B]();

    // Reinsertar elementos activos
    for (int i = 0; i < B_viejo; i++) {
        if (tablaVieja[i] != NULL && !eliminadoViejo[i]) {
            this->insertar(tablaVieja[i]->clave,
                           tablaVieja[i]->valor);
        }
    }

    // Liberar memoria vieja
    for (int i = 0; i < B_viejo; i++) {
        if (tablaVieja[i] != NULL) delete tablaVieja[i];
    }
    delete[] tablaVieja;
    delete[] eliminadoViejo;
}
¿Por qué reinsertar en lugar de copiar?
Con el nuevo B, los resultados de pos(key, i) cambian completamente (porque el módulo es % B). No podés copiar los elementos a las mismas posiciones — tenés que recalcular dónde va cada uno con la nueva función de posición.

Ejercicio 7Difícil
Explicá con tus palabras el problema del clustering primario en probing lineal y cómo lo mejora el probing cuadrático. ¿Por qué el doble hash es mejor que ambos?
Respuesta

Clustering primario: con probing lineal, cuando varios elementos colisionan en un bucket, forman una "cadena" de buckets consecutivos ocupados. Cualquier clave que mapee a cualquier bucket de esa cadena alarga la cadena, empeorando exponencialmente la situación.

Cuadrático: los saltos son 1, 4, 9, 16... así que los elementos no se agrupan consecutivamente. Pero si dos claves mapean al mismo bucket inicial, siguen exactamente la misma secuencia de probing → clustering secundario.

Doble hash: el salto depende de una segunda función de hash aplicada a la clave específica. Dos claves con el mismo bucket inicial tienen saltos distintos → no forman clusters. Es la distribución más uniforme de los tres.

Ejercicio 9Medio📋 Parcial May'23
En una tabla de hash cerrada, ¿por qué debemos marcar los elementos eliminados como "eliminados" (tombstone/lápida) y no simplemente como NULL? ¿Qué problema causaría marcarlo como NULL?
Respuesta

En hash cerrado con probing, cuando insertamos un elemento X que colisiona con Y, X queda en una posición posterior en la cadena de probing (ej: bucket+1, bucket+2...).

Si eliminamos Y y marcamos su bucket como NULL, al buscar X el algoritmo de búsqueda encuentra NULL en el bucket original y concluye que X no existe en la tabla — falso negativo. La búsqueda se interrumpe antes de llegar a X.

Con tombstone (lápida), el bucket queda marcado como "eliminado pero con historia". La búsqueda puede continuar el probing más allá de ese bucket. Al insertar nuevos elementos, un bucket con lápida puede reutilizarse.

Nota: En hash abierto (listas enlazadas) este problema no existe — simplemente se remueve el nodo de la lista, sin romper ninguna cadena de búsqueda.

Respuesta
  1. Determinista: para la misma clave siempre produce el mismo resultado. Sin esto, no podrías encontrar un elemento que ya insertaste.
  2. Distribución uniforme: distribuye las claves de manera pareja sobre todos los buckets. Minimiza colisiones y mantiene las operaciones cerca de O(1).

📅 Tema 3 — Heap Binario

1 ¿Qué es un Heap Binario?

Un heap binario es una estructura de datos que permite acceder, agregar y eliminar elementos de manera eficiente siguiendo un orden muy específico. Es la implementación más popular de una Cola de Prioridad.

📐 Definición formal

Un heap binario tiene exactamente dos propiedades que lo definen:

  • Propiedad estructural — determina la forma del árbol.
  • Propiedad de orden — determina la relación entre padre e hijos.
💡 ¿Para qué sirve?

El heap es la implementación más eficiente de una Cola de Prioridad (TAD). Garantiza acceso O(1) al elemento con mayor o menor prioridad, e inserción/eliminación en O(log n). Se usa en Dijkstra, Prim, HeapSort y cualquier algoritmo que necesite procesar elementos en orden de prioridad.

Sección 1 — ¿Qué es un Heap?
Heap binario = estructura de datos con 2 propiedades: estructural + de orden. Uso principal: implementar una Cola de Prioridad — siempre tenés el mínimo (o máximo) accesible en O(1). Operaciones base: insert O(log n), delete O(log n), peek O(1).

2 Las Dos Propiedades PARCIAL

Propiedad 1 — Estructural

El heap es un árbol binario completo: todos los niveles del árbol están completamente llenos, excepto posiblemente el último, que se llena de izquierda a derecha.

✅ Estructura válida vs ❌ inválida

VÁLIDO — último nivel de izq a der

1 / \ 3 5 / \ / 6 8 10

INVÁLIDO — hueco antes de llenar

1 / \ 3 5 / \ \ 6 8 10

El hueco en el hijo izquierdo de 5 viola la propiedad: el último nivel se llenó de derecha, no de izquierda.

Propiedad 2 — De Orden (Heap-Order)

Para cada nodo i, el valor del nodo es siempre mayor o igual que los de sus hijos (max-heap) o menor o igual (min-heap). No existe relación de orden entre hermanos.

Min-heap: padre ≤ hijos en todos los nodos

VÁLIDO

1 / \ 3 2 / \ / 6 8 4

INVÁLIDO — 2 menor que su padre 3

1 / \ 3 5 / \ / 6 2 10
❓ ¿Qué pasa si quitamos la propiedad estructural?

Si el heap ya no fuera un árbol casi completo, no podríamos usar un array para representarlo eficientemente. Necesitaríamos punteros explícitos (padre, hijo izquierdo, hijo derecho) como en un BST. Perdemos la principal ventaja del heap: representación compacta y acceso rápido a nodos mediante fórmulas simples.

Sección 2 — Las 2 Propiedades PARCIAL
Propiedad estructural: árbol binario completo. Todos los niveles llenos excepto el último, que se llena de izquierda a derecha sin huecos. Propiedad de orden: para cada nodo, padre ≤ hijos (min-heap) o padre ≥ hijos (max-heap). No hay orden entre hermanos.
¿Cuál permite el array? La propiedad estructural. Sin ella, necesitamos punteros — el heap pierde su ventaja principal. ¿Único heap posible con N elementos? No. Pueden existir múltiples heaps válidos con los mismos elementos (distintas posiciones en el último nivel).
🚨 Error frecuente — Propiedad de orden entre hermanos: el heap NO exige ninguna relación entre hermanos. Ver que 6 es hermano de 8 y 6 < 8 no viola ni cumple nada — ambas posiciones son válidas. El único requisito es padre ≤ hijos (min) o padre ≥ hijos (max). 🚨 Error frecuente — peek vs delete: peek es O(1) porque no modifica nada, simplemente devuelve heap[1]. Muchos estudiantes asumen que es O(log n) porque confunden peek con delete (que sí hace sink).

3 Min-Heap vs Max-Heap

La única diferencia es la dirección de la propiedad de orden. Estructura, operaciones y fórmulas de array son idénticas en ambos casos.

Min-Heap

Padre ≤ hijos. La raíz tiene el mínimo.

1 / \ 3 2 / \ / \ 7 5 4 9

Array: [1, 3, 2, 7, 5, 4, 9]

Max-Heap

Padre ≥ hijos. La raíz tiene el máximo.

9 / \ 7 4 / \ / \ 3 5 2 1

Array: [9, 7, 4, 3, 5, 2, 1]

Sección 3 — Min vs Max
Min-heap: padre ≤ hijos → raíz = mínimo. Max-heap: padre ≥ hijos → raíz = máximo. Solo cambia el comparador en swim y sink — todo lo demás (estructura, fórmulas, código) es idéntico.

4 Representación en Array PARCIAL

La propiedad estructural permite mapear el heap a un array recorriendo el árbol nivel a nivel, de izquierda a derecha. No se necesitan punteros.

Árbol → Array (lectura nivel por nivel)

1 ← idx 1 / \ 3 5 ← idx 2, 3 / \ / 6 8 10 ← idx 4, 5, 6
idx: [ 1 ][ 2 ][ 3 ][ 4 ][ 5 ][ 6 ] val: [ 1 ][ 3 ][ 5 ][ 6 ][ 8 ][ 10 ]
📐 Fórmulas — Base 1 (raíz en índice 1)
  • Hijo izquierdo(i) = 2i
  • Hijo derecho(i) = 2i + 1
  • Padre(i) = i / 2  (división entera)

El índice 0 del array queda sin usar. Esta es la notación del código del profesor.

📐 Fórmulas — Base 0 (raíz en índice 0)
  • Hijo izquierdo(i) = 2i + 1
  • Hijo derecho(i) = 2i + 2
  • Padre(i) = (i − 1) / 2  (división entera)
🔢 Múltiples heaps válidos con los mismos elementos

Para {1, 3, 5, 6, 8} existen varios min-heaps válidos — la propiedad de orden no dicta orden entre hermanos:

1 → [1, 3, 5, 6, 8] / \ 3 5 / \ 6 8
1 → [1, 3, 5, 8, 6] / \ 3 5 / \ 8 6

Ambos son min-heaps válidos. 6 y 8 son hijos de 3, y 3 ≤ 6 y 3 ≤ 8 en ambos casos.

🚨 Error frecuente en parcial — Mezclar base 0 con base 1

El error más común: usar la fórmula de base 1 (hijo = 2i) pero empezar el array en índice 0. Si el heap empieza en índice 0, la raíz está en heap[0] y la fórmula correcta es hijo_izq = 2i+1. Si usás 2i sobre un array base-0 obtenés el índice incorrecto.

Regla práctica: el código del profesor usa base 1 (la raíz está en heap[1], y heap[0] queda sin usar o como centinela). En el parcial, si no te aclaran, asumí base 1 y mencionalo explícitamente.

BASE 1: heap = [_, 9, 7, 4, 3, 6] ← índice 0 ignorado raíz = heap[1] = 9 hijo_izq(1) = heap[2·1] = heap[2] = 7 ✓ BASE 0: heap = [9, 7, 4, 3, 6] ← índice 0 es la raíz raíz = heap[0] = 9 hijo_izq(0) = heap[2·0+1] = heap[1] = 7 ✓ ❌ MEZCLA: raíz en heap[0] pero usando hijo = 2i → heap[0] = 9, hijo_izq = heap[0] = 9 (¡apunta a sí mismo!)
Sección 4 — Array y Fórmulas PARCIAL
BASE 1: hijo_izq(i) = 2i | hijo_der(i) = 2i+1 | padre(i) = i/2 BASE 0: hijo_izq(i) = 2i+1 | hijo_der(i) = 2i+2 | padre(i) = (i-1)/2 ¿Por qué base 1 es más cómoda? Las fórmulas son más simples. El índice 0 se desperdicia pero el código queda más limpio. ⚠️ Siempre decir qué base usás. Si empezás en índice 1 → usá 2i. Si empezás en índice 0 → usá 2i+1. Mezclarlos produce índices incorrectos sin que el código explote visiblemente.
¿Múltiples heaps con mismos elementos? Sí. No hay orden entre hermanos → distintas posiciones en el último nivel pueden ser todas válidas.

5 Operaciones: Swim y Sink PARCIAL

Insert — Swim (el elemento flota hacia arriba)

Para mantener la propiedad estructural, siempre se agrega el nuevo elemento al final del array (última posición del último nivel). Después se swim — se compara con su padre y sube hasta llegar a su posición correcta.

🚨 Errores frecuentes en parcial — Swim

1. Condición de parada: i > 1 vs i >= 1
En base 1, el swim termina cuando llegamos a la raíz (índice 1). La condición del while debe ser i > 1. Si escribís i >= 1, cuando i=1 calculás padre = 1/2 = 0 en división entera — y accedés a heap[0] que está vacío/indefinido en base 1. Loop potencialmente infinito o swap con basura.

2. Comparador: > vs >= para max-heap
En swim de max-heap la condición es heap[i] > heap[padre]. Si usás >= swapeás incluso cuando son iguales — no rompe el heap pero hace trabajo extra innecesario. En cambio, si usás < por error, jamás sube nada.

// Swim correcto (max-heap, base 1) while (i > 1 && heap[i] > heap[i/2]) { // ✓ i>1 y comparador > swap(heap[i], heap[i/2]); i = i / 2; } // ❌ Error clásico 1: i >= 1 → accede heap[0] cuando i=1 // ❌ Error clásico 2: heap[i] < heap[i/2] → condición invertida, nunca sube

Delete — Sink (el elemento se hunde hacia abajo)

Siempre se elimina la raíz (el máximo en un max-heap). Para mantener la propiedad estructural, se mueve el último elemento a la raíz y luego se sink — se compara con sus hijos y baja hasta su posición correcta.

🚨 Errores frecuentes en parcial — Sink

1. Comparar solo con el hijo izquierdo
En el sink hay que elegir el hijo más prioritario (el mayor en max-heap, el menor en min-heap) para hacer el swap. El error es comparar solo con el hijo izquierdo e ignorar el derecho. Si el derecho es mayor, terminás con el heap en estado inválido.

2. No chequear si el hijo derecho existe
El último nodo del último nivel puede no tener hijo derecho. Si calculás hijo_der = 2i+1 y directamente accedés a heap[2i+1] sin verificar que 2i+1 <= size, accedés a memoria fuera del heap.

3. > vs >= en el comparador
Igual que en swim — usá > (no >=) para no swapear cuando son iguales innecesariamente.

// Sink correcto (max-heap, base 1) while (2*i <= size) { // tiene al menos hijo izquierdo int j = 2*i; // j = hijo izquierdo if (j < size && heap[j] < heap[j+1]) // ¿existe hijo der y es mayor? j++; // j = hijo derecho (el mayor) if (heap[i] >= heap[j]) break; // ✓ padre ya es mayor o igual → listo swap(heap[i], heap[j]); i = j; } // ❌ Error 1: no chequear j < size antes de heap[j+1] → out of bounds // ❌ Error 2: swap con heap[2*i] sin comparar con heap[2*i+1] // ❌ Error 3: usar > en vez de >= en la condición de break → swap extra innecesario
Sección 5 — Swim y Sink PARCIAL
Insert → Swim (flota): agregar al final → mientras tenga padre y viola orden, swap con padre → subir. O(log n) peor caso. Delete → Sink (hunde): sacar raíz → mover último a raíz → mientras tenga hijos y viola orden, swap con el hijo más prioritario → bajar. O(log n).
¿Por qué mover el último a la raíz? Para preservar la propiedad estructural (árbol casi completo) sin dejar huecos. Peek: simplemente retornar heap[1] (base 1) o heap[0] (base 0) sin modificar nada. O(1).
🚨 Checklist de errores a evitar: SWIM: condición i > 1 (no >= 1) para no salirte del array en base 1 SWIM: comparador > (no <) — confundirlo invierte la dirección del float SINK: siempre elegir el MAYOR hijo (max-heap) — no automáticamente el izquierdo SINK: verificar que hijo_der = 2i+1 <= size antes de leer heap[2i+1] AMBOS: definir explícitamente si usás base 0 o base 1 al escribir código

📐 ¿Por qué insert es O(1) promedio?

En la mayoría de los casos, el número de swaps en el swim es pequeño. La probabilidad de que el elemento insertado tenga que recorrer toda la altura del árbol (log n) es baja. Matemáticamente, el número esperado de intercambios converge como una serie geométrica:

1 + 1/2 + 1/4 + 1/8 + ... = 2 → O(1)

Referencia del profesor: "Average Case Analysis of Heap Building by Repeated Insertion".

💡 ¿Por qué search es O(n)?

El heap solo garantiza que el mínimo (o máximo) esté en la raíz. No existe relación de orden entre hermanos — un elemento cualquiera puede estar en cualquier posición. Hay que recorrer todo el array en el peor caso.

Patrón de respuesta "justificá el orden": siempre decir (1) qué operación hace el trabajo (swim/sink), (2) cuántos pasos puede hacer como máximo (altura = log n), (3) para el promedio mencionar la serie geométrica.

Responder solo O(1) sin aclarar que es promedio está incompleto y puede costarte puntos. 🚨 Error frecuente — Build-Heap: Build-Heap no es insertar n elementos uno a uno (eso sería O(n log n)). Es aplicar siftDown desde i=n/2 hasta i=1 (base 1) — O(n). El índice de inicio es n/2, no n. Los nodos de índice > n/2 son hojas y no necesitan siftDown.

7 Cola de Prioridad (TAD)

Una Cola de Prioridad es un TAD que almacena elementos con una prioridad asociada. Sus operaciones son:

  • enqueue: inserta un elemento con una prioridad.
  • dequeue: elimina y retorna el elemento con la prioridad más alta.
  • peek: retorna el elemento con la prioridad más alta sin eliminarlo.

Comparación de implementaciones

El heap binario es la implementación más eficiente para una Cola de Prioridad. Estas son las alternativas:

Implementaciónenqueuedequeuepeek
Lista no ordenada O(1) O(n) O(n)
Lista ordenada O(n) O(1) O(1)
Árbol AVL O(log n) O(log n) O(log n)
Heap binario ✓ O(log n)* O(log n) O(1)

* O(1) promedio en insert.

⚠️ Cola de Prioridad ≠ Heap Binario

La Cola de Prioridad es el TAD (qué hace). El Heap Binario es la implementación (cómo lo hace). Es normal confundirlos porque tienen métodos similares: enqueue → insert, dequeue → delete, peek → peek.

Sección 7 — Cola de Prioridad
CP TAD: enqueue (insertar con prioridad), dequeue (sacar el de mayor prioridad), peek (ver sin sacar). Heap gana porque: peek O(1), enqueue O(log n) / O(1) prom, dequeue O(log n). Mejor que lista ordenada y lista no ordenada en el conjunto de las tres operaciones. AVL vs Heap: el AVL es mejor para búsquedas generales. El Heap es mejor para acceso siempre al mínimo/máximo.

8 Cola de Prioridad Extendida: Heap + Hash PARCIAL

Una Cola de Prioridad Extendida agrega dos operaciones a la CP normal:

  • changePriority: cambia la prioridad de un elemento ya insertado.
  • delete: elimina un elemento cualquiera (no solo la raíz).
😵 El problema: solo con un heap

Para hacer delete(x) o changePriority(x), primero hay que encontrar x en el heap. Como search es O(n), ambas operaciones quedan en O(n). Ineficiente.

📐 La solución: Heap + Tabla de Hash

La tabla de hash mapea elemento → posición en el heap. Así encontramos cualquier elemento en O(1). Luego swim o sink en O(log n).

  • delete(x): hash[x] → posición i en O(1) → swap con último → siftDown → O(log n).
  • changePriority(x): hash[x] → posición i en O(1) → swim o sink según corresponda → O(log n).

⚠️ Hay que modificar insert y delete para que también actualicen la tabla de hash cada vez que un swap cambia la posición de un elemento.

🤝 Lo mejor de los dos mundos

El profesor lo dice explícitamente: la combinación Heap + Hash te da lo mejor de cada estructura porque cada una compensa exactamente la debilidad de la otra.

HEAP — lo que sabe hacer

✅ mantener prioridades (swim/sink) ✅ peek O(1) — raíz siempre disponible ✅ insert/deleteMin en O(log n) ❌ encontrar un elemento arbitrario → O(n)
+

HASH — lo que sabe hacer

✅ encontrar cualquier elemento → O(1) ✅ buscar/insertar/eliminar en O(1) ❌ no tiene concepto de orden/prioridad ❌ no sabe quién es el mínimo/máximo
=

HEAP + HASH — resultado

✅ peek O(1) — sigue siendo la raíz ✅ insert O(log n) ✅ deleteMin O(log n) ✅ delete any O(log n) ← hash resuelve el O(n) ✅ changePriority O(log n) ← hash resuelve el O(n)

La hash guarda elemento → posición en el heap. Con eso, encontrar cualquier elemento es O(1) y después el heap se encarga del reordenamiento (swim/sink) en O(log n). La única complejidad extra es mantener el hash sincronizado: cada vez que swim o sink hace un swap, hay que actualizar las dos entradas en el hash.

💡 ¿Para qué usaríamos heap + hash? (pregunta directa de parcial)

Para implementar una Cola de Prioridad Extendida. El heap mantiene el orden de prioridades (O(log n) para swim/sink). La hash permite encontrar cualquier elemento en O(1), convirtiendo delete-any y changePriority de O(n) a O(log n). Es la combinación ideal para algoritmos como Dijkstra que necesitan actualizar prioridades frecuentemente.

🔧 Delete-any sin estructuras auxiliares (pregunta directa May 2023)

El examen preguntó: "¿Cómo implementar un heap que permita eliminar elementos no solo desde la raíz, sin usar estructuras de datos auxiliares?"

Sin hash table, solo podemos escanear el array para encontrar el elemento. El truco es lo que hacemos después de encontrarlo:

Algoritmo delete(x) sin auxiliares:

  1. Escanear el array hasta encontrar x → posición i. O(n)
  2. Swap: heap[i] ↔ heap[size]. Después size--.
  3. Llamar swim(i). Después llamar sink(i). Siempre los dos, en ese orden. Una de las dos no hará nada.

💡 ¿Por qué siempre swim y luego sink, sin pensar?

El elemento que quedó en posición i puede estar fuera de lugar hacia arriba o hacia abajo — no sabemos cuál. En vez de razonar el caso, el profesor dice: siempre flotar, luego hundir. Si el elemento tiene que subir, swim lo mueve y sink no toca nada (ya cumple la condición con sus hijos). Si tiene que bajar, swim no hace nada (ya cumple con su padre) y sink lo mueve. Si ya está en su lugar, ambos salen inmediatamente. El resultado es correcto en los tres casos — cero razonamiento extra requerido.

Complejidad final:

scan: O(n) swim o sink: O(log n) ──────────────────── total: O(n)

¿Es mejor con hash?

sin hash: O(n) ← scan domina con hash: O(log n) ← hash O(1) + sink O(log n) la diferencia importa si delete-any es frecuente
Sección 8 — Cola de Prioridad Extendida PARCIAL
CP Normal: insert, findMin, deleteMin. CP Extendida agrega: changePriority(x) y delete(x) para cualquier x.
Solo heap, sin aux: scan O(n) → swap con último → swim(i) → sink(i). Total O(n). Heap + Hash: hash[x] da posición en O(1) → swim(i) → sink(i). Total O(log n). Hay que actualizar el hash en cada swap.
Regla del profe — siempre flotar, luego hundir: después de colocar un elemento en posición i (ya sea por delete-any o changePriority), siempre llamar swim(i) y después sink(i), sin pensar. Si tiene que subir, swim lo mueve y sink no hace nada. Si tiene que bajar, swim no hace nada y sink lo mueve. Si ya está en su lugar, ambos terminan en O(1). Sin condicionales, sin razonamiento de caso — siempre correcto.
Patrón de respuesta en examen: siempre responder las 3 partes — (1) qué operaciones agrega, (2) qué orden tienen solo con heap, (3) cómo mejorar con hash y a qué orden llegás.

9 Código del Profesor (Max-Heap, base 1)

La implementación del profesor usa templates C++, base 1 (índice 0 sin usar), y los métodos swim (sube) y sink (baja).

Clase BinaryHeap

template <typename T>
class BinaryHeap {
public:
    BinaryHeap(int capacity) : size(0), capacity(capacity) {
        heap = new T[capacity + 1]; // +1 porque base 1 (índice 0 sin usar)
    }
    ~BinaryHeap() { delete[] heap; }

    void insert(T value) {
        if (size == capacity) { /* heap lleno */ return; }
        size++;
        heap[size] = value;
        swim(size);       // el nuevo elemento "flota" hacia arriba
    }

    T removeMax() {
        T maxValue = heap[1];       // raíz = máximo
        swap(1, size);               // último elemento ocupa la raíz
        size--;
        sink(1);                    // la raíz "se hunde" hasta su lugar
        return maxValue;
    }

    bool isEmpty() { return size == 0; }

private:
    T* heap;
    int size;
    int capacity;

Swim (sube) y Sink (baja)

    void swim(int pos) {
        if (pos > 1) {
            int posPadre = pos / 2;
            if (heap[posPadre] < heap[pos]) {   // padre menor → swap (max-heap)
                swap(pos, posPadre);
                swim(posPadre);              // recursión — sigue subiendo
            }
        }
    }

    void sink(int pos) {
        int posHijoIzq = 2 * pos;
        if (posHijoIzq <= size) {
            int posHijoMayor = posHijoIzq;
            int posHijoDer = posHijoIzq + 1;
            if (posHijoDer <= size && heap[posHijoDer] > heap[posHijoIzq])
                posHijoMayor = posHijoDer;   // elegir el hijo mayor
            if (heap[posHijoMayor] > heap[pos]) {
                swap(pos, posHijoMayor);
                sink(posHijoMayor);          // recursión — sigue bajando
            }
        }
    }

    void swap(int i, int j) {
        T temp = heap[i]; heap[i] = heap[j]; heap[j] = temp;
    }
};
Sección 9 — Código Clave
swim: if pos > 1 → posPadre = pos/2 → si padre < yo → swap → swim(posPadre). (max-heap, recursivo) sink: if hijoIzq <= size → elegir hijo mayor → si hijo mayor > yo → swap → sink(posHijoMayor). (recursivo)
Base 1: heap[1] = raíz. heap = new T[capacity + 1] para tener índices 1..capacity. Desventaja del array: tamaño fijo — no se puede redimensionar eficientemente sin copiar todo.

Si tenés 3 elementos, están en arr[0], arr[1], arr[2] y sigLibre == 3. Es el índice donde iría el próximo elemento. Por eso estaVacio() es sigLibre == 0 y estaLleno() es sigLibre == capacidad.

¿Por qué cada línea?

  • sigLibre = 0 — arrancamos vacío. sigLibre es 0 porque no hay ningún elemento, y la posición 0 es donde iría el primero.

Flotar — sube un elemento

Compara con el padre. Si el hijo es menor que el padre (min-heap), intercambia y sigue subiendo.

template <class T>
void MinHeap<T>::flotar(int pos) {
    if (pos > 0) {
        int posPadre = padre(pos);
        if (arr[pos] < arr[posPadre]) {
            intercambiar(pos, posPadre);
            flotar(posPadre);  // llamada recursiva — sigue subiendo
        }
    }
}

¿Por qué cada línea?

  • if (pos > 0) — es el caso base de la recursión. Si pos == 0 ya estamos en la raíz, no hay padre con quién comparar, así que la función no hace nada y la recursión termina. En base 0, la raíz es el índice 0. En base 1 sería pos > 1.
  • int posPadre = padre(pos) — calculamos la posición del padre una sola vez y la guardamos. ¿Por qué? Porque la vamos a usar dos veces (para comparar y para intercambiar). Llamar a padre(pos) cada vez no sería un error, pero guardarla en una variable es más claro y evita recalcular.
  • if (arr[pos] < arr[posPadre]) — esta es la condición de violación del min-heap. Si el hijo es menor que su padre, la propiedad de orden está rota: en un min-heap, el padre siempre tiene que ser ≤ que sus hijos. Si no se cumple, hay que arreglar.
  • intercambiar(pos, posPadre) — arreglamos la violación local: ponemos al menor arriba. Pero ojo: al mover el elemento al lugar del padre, ahora puede estar violando la propiedad con el abuelo.
  • flotar(posPadre) — por eso llamamos recursivamente. El elemento ahora está en la posición del padre, y necesitamos verificar si también es menor que el abuelo, y así sucesivamente. La recursión termina cuando: (a) llegamos a la raíz (pos == 0), o (b) el padre ya es menor o igual (arr[pos] < arr[posPadre] es falso).
🚨 ¿Por qué no necesitamos un else?

Si arr[pos] < arr[posPadre] es falso, significa que el padre ya es menor o igual que el hijo — la propiedad de orden se cumple. No hay nada que arreglar, ni en este nivel ni más arriba (porque todos los ancestros ya cumplían antes). La función simplemente no entra al if y termina.

Hundir — baja un elemento

Compara con los hijos. Intercambia con el menor de los dos hijos y sigue bajando.

template <class T>
void MinHeap<T>::hundir(int pos) {
    int posHijoIzq = hijoIzq(pos);
    if (posHijoIzq < sigLibre) {              // si tiene al menos hijo izquierdo
        int posHijoDer = hijoDer(pos);
        int posHijoMenor = posHijoIzq;

        // Si existe hijo derecho Y es menor que el izquierdo
        if (posHijoDer < sigLibre && arr[posHijoDer] < arr[posHijoIzq])
            posHijoMenor = posHijoDer;

        // Si el hijo menor es más chico que el padre → intercambiar y seguir
        if (arr[posHijoMenor] < arr[pos]) {
            intercambiar(pos, posHijoMenor);
            hundir(posHijoMenor);             // llamada recursiva — sigue bajando
        }
    }
}

¿Por qué cada línea?

  • int posHijoIzq = hijoIzq(pos) — calculamos dónde estaría el hijo izquierdo. Lo guardamos porque lo vamos a usar varias veces.
  • if (posHijoIzq < sigLibre) — caso base de la recursión. sigLibre es la primera posición vacía. Si el hijo izquierdo cae en sigLibre o más allá, ese hijo no existe — el nodo es una hoja y no puede bajar más. ¿Por qué solo chequeamos el izquierdo? Porque el heap se llena de izquierda a derecha: si no hay hijo izquierdo, tampoco puede haber derecho.
  • int posHijoDer = hijoDer(pos) — calculamos la posición del hijo derecho. Puede o no existir (por eso verificamos antes de usarlo).
  • int posHijoMenor = posHijoIzq — asumimos que el izquierdo es el menor. Es nuestra hipótesis inicial que después corregimos si es necesario.
  • if (posHijoDer < sigLibre && ...) — primero chequeamos que el hijo derecho exista (posHijoDer < sigLibre). Si no existe, ni evaluamos la segunda condición (short-circuit). Si existe Y es menor que el izquierdo, actualizamos posHijoMenor. ¿Por qué elegimos el menor? Porque si intercambiamos con el mayor, el menor quedaría como hijo de un nodo más grande que él — crearíamos una nueva violación.
  • if (arr[posHijoMenor] < arr[pos]) — verificamos si hay violación: ¿el hijo menor es más chico que el padre? Si sí, la propiedad de orden está rota. Si no, el padre ya es menor o igual que ambos hijos — todo está bien y la recursión termina.
  • intercambiar(pos, posHijoMenor) — arreglamos la violación local: el menor sube, el padre baja.
  • hundir(posHijoMenor) — el elemento que bajó ahora puede estar violando la propiedad con sus hijos (los nietos del nodo original). Llamamos recursivamente para seguir bajándolo hasta que encuentre su lugar o llegue a una hoja.
🚨 ¿Por qué el menor de los hijos y no cualquiera?

Ejemplo: padre = 8, hijo izq = 3, hijo der = 5. Si intercambiamos 8 con 5 (el mayor), queda: padre = 5, hijo izq = 3. Pero 3 < 5 — ¡rompimos la propiedad! Si intercambiamos 8 con 3 (el menor), queda: padre = 3, hijo izq = 8, hijo der = 5. Tanto 8 como 5 son mayores que 3 — correcto. Siempre intercambiar con el menor garantiza que el nuevo padre es ≤ que ambos hijos.

Insertar

template <class T>
void MinHeap<T>::insertar(T el) {
    assert(!estaLleno());
    arr[sigLibre++] = el;      // poner al final, avanzar sigLibre
    flotar(sigLibre - 1);      // flotar el elemento que acabamos de poner
}

¿Por qué cada línea?

  • assert(!estaLleno()) — si el heap está lleno, no hay espacio en el array. Intentar escribir en arr[sigLibre] cuando sigLibre == capacidad sería escribir fuera de los límites del array (undefined behavior en C++). El assert detiene el programa con un mensaje claro en vez de dejarlo romperse silenciosamente.
  • arr[sigLibre++] = el — ponemos el elemento nuevo en la primera posición libre (al final del heap). ¿Por qué al final? Porque es el único lugar donde podemos ponerlo sin romper la propiedad estructural (árbol binario completo que se llena de izquierda a derecha). El ++ post-incremento significa: usá el valor actual de sigLibre como índice, y después incrementalo en 1.
  • flotar(sigLibre - 1) — acabamos de poner un elemento al final. La propiedad estructural se mantiene, pero la propiedad de orden puede estar rota (el nuevo elemento puede ser menor que su padre). Flotar lo sube hasta su lugar correcto. Usamos sigLibre - 1 porque el ++ ya avanzó sigLibre — el elemento quedó una posición antes.

Obtener el mínimo

template <class T>
T MinHeap<T>::obtenerMin() {
    assert(!estaVacio());
    return arr[0];  // la raíz siempre es el mínimo (base 0)
}

Eliminar el mínimo

template <class T>
T MinHeap<T>::eliminarMin() {
    assert(!estaVacio());
    T minimo = arr[0];              // guardar la raíz
    arr[0] = arr[--sigLibre];        // último elemento ocupa la raíz
    hundir(0);                       // hundir la raíz hasta su lugar
    return minimo;
}

¿Por qué cada línea?

  • assert(!estaVacio()) — no podés eliminar de un heap vacío. Si sigLibre == 0, no hay elementos — arr[0] es basura.
  • T minimo = arr[0] — guardamos el mínimo antes de destruirlo, porque lo necesitamos devolver al final.
  • arr[0] = arr[--sigLibre] — ¿por qué no simplemente borramos arr[0]? Porque dejaría un "hueco" en la raíz, rompiendo la propiedad estructural. En cambio, tomamos el último elemento del heap y lo ponemos en la raíz. El --sigLibre pre-decremento reduce el tamaño en 1 primero, y después usa ese valor como índice del último elemento. Así el heap tiene un elemento menos y no hay huecos.
  • hundir(0) — el elemento que pusimos en la raíz probablemente no es el mínimo de todo el heap. Estará fuera de lugar — sus hijos pueden ser menores que él. hundir lo baja comparándolo con sus hijos hasta que encuentre su posición correcta o llegue a una hoja.
  • return minimo — devolvemos el valor que guardamos al principio, no arr[0] (que ahora tiene otro valor).
💡 ¿Por qué no borrar la raíz directamente y mover todo un lugar?

Porque mover todos los elementos del array un lugar costaría O(n). En cambio, swap raíz ↔ último + hundir cuesta O(log n) — solo baja por una rama del árbol (la altura es log n).

Ejercicio 1Fácil
Describí las dos propiedades del heap binario. ¿Cuál de ellas permite representarlo como array? ¿Qué pasa si la quitamos?
Respuesta

Propiedad 1 — Estructural: árbol binario completo. Todos los niveles llenos excepto el último, que se llena de izquierda a derecha.

Propiedad 2 — De orden: para todo nodo, padre ≤ hijos (min-heap) o padre ≥ hijos (max-heap). No hay orden entre hermanos.

Ejercicio 8Difícil
Explicá cómo encontrar los K elementos más grandes de un array de N elementos usando un min-heap de tamaño K. ¿Cuál es la complejidad total y por qué es mejor que ordenar el array?
Respuesta

Algoritmo: mantener un min-heap de exactamente K elementos.

  1. Insertar los primeros K elementos en el min-heap. O(K log K).
  2. Para cada uno de los N−K elementos restantes: si el elemento es mayor que la raíz del min-heap (el menor de los K candidatos actuales) → reemplazar la raíz y hacer sink. O(log K) por elemento.
  3. Al final, los K elementos del min-heap son los K más grandes del array.

Complejidad total: O(N log K). Para cada uno de los N elementos se hace como máximo un sink de tamaño K.

¿Mejor que ordenar? Sí, cuando K << N. Ordenar es O(N log N) siempre. Con esta técnica, si K=10 y N=1.000.000: O(N log 10) ≈ O(3.3N) vs O(N × 20) = O(20N). Mucho mejor.

Ejercicio 9Medio📋 Parcial May'23
¿Cómo implementarías un heap que permita la eliminación de elementos no solo desde la raíz? (no usar estructuras de datos auxiliares)

Describí el algoritmo paso a paso y justificá su complejidad.
Respuesta

Sin estructuras auxiliares, el único camino es escanear el array para encontrar el elemento. Luego:

  1. Encontrar el elemento x en el array → posición i. O(n).
  2. Swap heap[i] con heap[size] (el último elemento).
  3. size-- (el elemento queda fuera del heap lógicamente).
  4. Llamar swim(i). Luego llamar sink(i). Siempre ambos, siempre en ese orden — sin razonar el caso. Si el elemento tiene que subir, swim lo mueve y sink sale inmediatamente. Si tiene que bajar, swim sale inmediatamente y sink lo mueve. Si ya está en su lugar, ambos terminan en O(1).

Complejidad: O(n) para el scan + O(log n) para swim/sink = O(n) total. El scan domina.

¿Cuándo conviene agregar una hash table? Si delete-any se llama frecuentemente y la constante importa — la hash reduce el delete-any a O(log n), pero agrega O(n) de memoria y overhead en cada swap.

Ejercicio 11Medio📋 Parcial May'23
¿Cuál es la complejidad de tiempo (caso promedio y peor caso) de las operaciones de inserción, búsqueda y eliminación en un Heap? Tomá dos de ellas y explicá la razón de su orden.
Respuesta
Operación Promedio Peor caso
Insert O(1) O(log n)
Search O(n) O(n)
Delete (raíz) O(log n) O(log n)

Insert — peor caso O(log n): al insertar, el nuevo elemento se agrega al final y hace swim hacia arriba. En el peor caso recorre toda la altura del árbol, que es log₂(n) para un árbol casi completo de n nodos. Cada nivel es un swap, entonces el peor caso es O(log n).

Insert — promedio O(1): la probabilidad de que el elemento tenga que recorrer toda la altura es baja. Matemáticamente, los intercambios esperados forman una serie geométrica: 1 + 1/2 + 1/4 + 1/8 + ... = 2, que es una constante → O(1) promedio.

Search — O(n): el heap solo garantiza que el mínimo (o máximo) esté en la raíz. No hay ninguna relación de orden entre nodos del mismo nivel o entre hermanos. Para encontrar un elemento cualquiera, en el peor caso hay que revisar todo el array → O(n).

Espera — revisemos todos: pos1=1≤pos2=3✓, pos1=1≤pos3=2✓, pos2=3≤pos4=7✓, pos2=3≤pos5=5✓, pos3=2≤pos6=8✓, pos3=2≤pos7=4✓. En realidad C también es válido ✅. ¡Ojo — el heap no requiere orden entre pos6 y pos7, solo entre padre e hijo!

📅 Tema 4 — Grafos

1 ¿Qué es un Grafo?

Un grafo es un tipo abstracto de dato que consiste en un conjunto de nodos (vértices) y aristas que definen relaciones entre los nodos.

📐 Definición formal
  • Nodo / Vértice: Representa un objeto o entidad.
  • Arista: Representa una relación entre dos nodos.

Denotamos V a la cantidad de vértices y A (o E) a la cantidad de aristas.

💡 Dos propiedades básicas de las aristas
  • Ordenación (direccionalidad): ¿la arista tiene dirección o es bidireccional?
  • Ponderación: ¿la arista tiene un peso/costo asociado?

4 combinaciones posibles

Dirigido

Las aristas tienen dirección. Ej: Instagram (seguir a alguien).

No dirigido

Las aristas son bidireccionales. Ej: Facebook (amistad).

Ponderado

Las aristas tienen peso. Ej: mapa de rutas (distancia entre ciudades).

No ponderado

Las aristas no tienen peso. Ej: árbol genealógico.

Sección 1 — ¿Qué es un Grafo?
Grafo = conjunto de vértices (V) + aristas (A) que definen relaciones. Aristas se clasifican por: direccionalidad (dirigido/no dirigido) y ponderación (con/sin peso).

2 Clasificaciones

Densidad

Relación entre la cantidad de aristas que tiene y la cantidad máxima posible.

Denso

Cantidad de aristas cercana al máximo. O(A) ≈ O(V²).

Disperso

Cantidad de aristas mucho menor al máximo. O(A) << O(V²).

📐 Aristas máximas posibles

No dirigido: V(V−1)/2

Cada par de nodos tiene como máximo 1 arista.

Dirigido: V(V−1)

Cada par puede tener 2 aristas (ida y vuelta).

🚨 Regla del profesor

Grafo disperso → Lista de adyacencia

Grafo denso → Matriz de adyacencia

Esto no es una preferencia — es la elección correcta según la estructura del grafo. A continuación, por qué.

¿Por qué disperso → lista de adyacencia?

El problema de la matriz con grafos dispersos: desperdicio de memoria

Imaginá un grafo con 10.000 nodos (una red social) donde cada persona tiene en promedio 10 amigos.

Aristas reales: ~50.000 (10.000 × 10 / 2) Aristas posibles: 10.000 × 9.999 / 2 = ~50.000.000 Con MATRIZ de adyacencia: Almacenás: 10.000 × 10.000 = 100.000.000 celdas Celdas con información útil: 50.000 Celdas vacías (INF o 0): 99.950.000 → ¡99.95% de la memoria está desperdiciada! 🗑️ Con LISTA de adyacencia: Almacenás: 10.000 listas + 100.000 nodos en total (2 por arista) → Solo guardás lo que existe. Cero desperdicio.

Pero no es solo memoria — también es velocidad

Si querés recorrer el grafo (BFS/DFS), necesitás saber "¿quiénes son los vecinos de este nodo?"

Nodo 42 tiene 10 vecinos. Con LISTA: recorrés su lista → 10 operaciones ✓ Con MATRIZ: recorrés toda la fila 42 → 10.000 operaciones De esas 10.000, solo 10 son vecinos reales. Las otras 9.990 son "no hay arista" → trabajo al pedo.

Resultado: BFS/DFS con lista = O(V+A). Con matriz = O(V²). En un grafo disperso donde A ≈ V, la diferencia es O(V) vs O(V²) — abismal.

¿Por qué denso → matriz de adyacencia?

Cuando el grafo es denso, la matriz ya no desperdicia

Un grafo denso tiene A ≈ V² aristas — casi todas las conexiones posibles existen.

Grafo denso: V=1000, A≈500.000 Con MATRIZ: Espacio: 1000 × 1000 = 1.000.000 celdas Celdas con info: ~500.000 → 50% llenas. Razonable. Con LISTA: Espacio: 1000 listas + 1.000.000 nodos (2 × 500.000) + overhead de punteros, nodos de lista, etc. → Puede usar MÁS memoria que la matriz por el overhead.

Y la matriz tiene ventajas que la lista no puede igualar

  • ¿Existe la arista u→v? Matriz: adjMatrix[u][v] != INF → O(1). Lista: recorrer la lista de u buscando v → O(grado(u)).
  • Eliminar una arista: Matriz: adjMatrix[u][v] = INF → O(1). Lista: buscar y eliminar el nodo de la lista → O(grado(u)).
  • Floyd-Warshall: el algoritmo trabaja directamente sobre una matriz. Con lista tendrías que convertir primero.

¿Y la velocidad de recorrido en un grafo denso?

Cuando A ≈ V², ambas implementaciones dan la misma complejidad:

BFS/DFS con lista: O(V + A) = O(V + V²) = O(V²) BFS/DFS con matriz: O(V²) → Empate en velocidad de recorrido. → Pero la matriz gana en consultas de aristas O(1) y en eliminar aristas O(1).

⚠️ En la práctica

La mayoría de los grafos reales son dispersos — redes sociales, internet, mapas de rutas, dependencias de software. Cada nodo se conecta con pocos otros en relación al total. Por eso la lista de adyacencia es la implementación por defecto en casi todos los casos.

Los grafos densos aparecen en problemas específicos: matrices de distancias entre ciudades, correlaciones entre variables, problemas "todos contra todos".

Contiene al menos un ciclo — un camino que comienza y termina en el mismo nodo.

Acíclico

Conexidad

Grafo no dirigido

Existe un camino entre cualquier par de nodos.

No conexo

Existen nodos que no están conectados.

Grafo dirigido

No conexo

Hay nodos desconectados sin importar la dirección.

¿Cómo determinar la conexidad con BFS?

📐 La idea clave del profesor

Después de hacer un BFS desde un nodo, el array encolado[] te dice exactamente a qué nodos pudiste llegar. Si todos son true, significa que desde ese nodo hay camino a todos los demás.

Grafo no dirigido — ¿Es conexo?

Un solo BFS desde cualquier nodo es suficiente.

1. Hacer BFS desde cualquier nodo (ej: nodo 0) 2. Mirar el array encolado[] 3. ¿Todos son true? → SÍ: el grafo es conexo (hay camino entre cualquier par) → NO: el grafo NO es conexo (hay nodos inalcanzables)

¿Por qué basta con un solo BFS? Porque en un grafo no dirigido, si podés ir de A a B, también podés ir de B a A (las aristas van en ambas direcciones). Entonces si desde un nodo llegás a todos, todos llegan a todos.

Complejidad: O(V+A) — un solo BFS.

Grafo dirigido — ¿Es fuertemente conexo?

Un BFS no alcanza. ¿Por qué? Porque que A llegue a B no significa que B llegue a A (las aristas tienen dirección). Necesitás verificar que desde cada nodo se llega a todos los demás.

1. Para CADA nodo i (de 0 a V-1): a. Hacer BFS desde i b. Mirar el array encolado[] c. ¿Todos son true? → NO en algún BFS: NO es fuertemente conexo. Parar. 2. Si TODOS los V BFS dieron todos true: → El grafo es fuertemente conexo

¿Por qué V veces? Porque "fuertemente conexo" requiere que haya camino de cualquier nodo a cualquier otro. Un BFS desde el nodo 0 te dice que 0 llega a todos, pero no que todos llegan a 0. Hacerlo V veces verifica todas las direcciones.

Complejidad: O(V × (V+A)) — V veces un BFS.

Grafo dirigido — ¿Es débilmente conexo?

"Débilmente conexo" significa que sería conexo si ignoramos las direcciones de las aristas. Para verificar:

1. Construir el "grafo subyacente no dirigido": para cada arista u→v, agregar también v→u 2. Hacer UN solo BFS en ese grafo no dirigido 3. ¿Todos encolado[] son true? → SÍ: débilmente conexo → NO: no conexo

Complejidad: O(V+A) — construir el grafo subyacente + un BFS.

¿Cuántas componentes conexas hay? (grafo no dirigido)

Una componente conexa es un grupo de nodos donde todos se conectan entre sí pero no con nodos de otros grupos.

contador = 0 Para cada nodo i de 0 a V-1: si encolado[i] == false: // nodo no alcanzado aún hacer BFS desde i // alcanza toda su componente contador++ // encontramos una componente nueva return contador

¿Por qué funciona? El primer BFS marca como encolado = true todos los nodos de la primera componente. Cuando seguimos iterando, los nodos ya encolados se saltan. El próximo nodo con encolado == false pertenece a otra componente — hacemos BFS desde ahí y marcamos toda esa componente. Cada BFS nuevo = una componente nueva.

Complejidad: O(V+A) total — aunque hacemos varios BFS, cada nodo y arista se procesa exactamente una vez en total (el array encolado es global, no se reinicia entre BFS).

💡 clasificarGrafo() — todo junto (pregunta de parcial May 2024)

Para clasificar un grafo dirigido como fuertemente conexo / débilmente conexo / no conexo:

  1. Verificar fuertemente conexo: hacer BFS desde cada nodo. Si todos los encolado[] dan true en cada BFS → fuertemente conexo. Si falla alguno, pasar al paso 2.
  2. Verificar débilmente conexo: construir el grafo subyacente no dirigido. Hacer BFS desde cualquier nodo. Si todos encolado[] son true → débilmente conexo.
  3. Si ni siquiera débilmente conexo: no conexo.
Sección 2 — Clasificaciones PARCIAL
5 formas de clasificar: Ordenación, Ponderación, Densidad, Ciclos, Conexidad. Denso vs Disperso: determina la implementación (matriz vs lista). DAG: Directed Acyclic Graph — necesario para orden topológico.
Cómo usar BFS + encolado[] para conexidad: No dirigido conexo: 1 BFS, todos encolado[] == true → O(V+A). Dirigido fuertemente conexo: V BFS, todos deben dar todos true → O(V(V+A)). Dirigido débilmente conexo: construir grafo subyacente (ignorar direcciones) + 1 BFS → O(V+A). Componentes conexas: BFS iterativo con encolado[] global. Cada BFS nuevo = una componente más → O(V+A) total.

3 Implementaciones

Dos formas de representar un grafo en memoria, cada una con sus ventajas.

Lista de Adyacencia

Un array de listas. Cada celda representa un nodo, cada lista contiene sus vecinos. Eficiente en memoria para grafos dispersos.

En no dirigido, cada arista se almacena dos veces (una en cada lista).

Matriz de Adyacencia

Matriz V×V donde cada elemento indica si hay arista (y su peso). Acceso O(1) a cualquier arista. Eficiente para grafos densos.

En no dirigido, la matriz es simétrica. Sin arista = INF (ponderado) o 0 (no ponderado).

Struct Arista

struct Arista {
    int origen;
    int destino;
    int peso;

    Arista() {}
    Arista(int _origen, int _destino, int _peso)
        : origen(_origen), destino(_destino), peso(_peso) {}
};

Código — Lista de Adyacencia (notación del profesor)

class ListaAdyacencia : Grafo {
private:
    NodoLista<Arista>** listaNodos;  // array de punteros a listas enlazadas
    int V;                           // cantidad de vértices
    int A;                           // cantidad de aristas
    bool esPonderado;
    bool esDirigido;

public:
    ListaAdyacencia(int _V, bool _esDirigido, bool _esPonderado) {
        V = _V;
        A = 0;
        esDirigido = _esDirigido;
        esPonderado = _esPonderado;
        listaNodos = new NodoLista<Arista>*[V + 1];
        for (int i = 0; i <= V; i++) {
            listaNodos[i] = NULL;
        }
    }

    void aniadirArista(int origen, int destino, int peso = 1) {
        assert(esPonderado || peso == 1);  // si no es ponderado, peso debe ser 1
        Arista a(origen, destino, peso);
        listaNodos[origen] = new NodoLista<Arista>(a, listaNodos[origen]);
        A++;

        if (!esDirigido) {
            Arista aInversa(destino, origen, peso);
            listaNodos[destino] = new NodoLista<Arista>(aInversa, listaNodos[destino]);
        }
    }

    NodoLista<Arista>* adyacentesA(int origen) {
        NodoLista<Arista>* ret = NULL;
        NodoLista<Arista>* aux = listaNodos[origen];
        while (aux != NULL) {
            ret = new NodoLista<Arista>(aux->dato, ret);
            aux = aux->siguiente;
        }
        return ret;
    }
};

¿Por qué adyacentesA devuelve una copia?

  • ret = NULL — arrancamos con una lista vacía nueva.
  • aux = listaNodos[origen] — aux recorre la lista original sin modificarla.
  • ret = new NodoLista<Arista>(aux->dato, ret) — para cada arista de la lista original, creamos un nodo nuevo y lo insertamos al inicio de ret. Es el mismo patrón que aniadirArista: insertar al inicio en O(1).
  • aux = aux->siguiente — avanzamos al siguiente nodo de la lista original.
  • ¿Por qué una copia y no devolver listaNodos[origen] directamente? Porque si devolvés el puntero interno, el que lo recibe podría modificar la lista (borrar nodos, cambiar datos) y rompería el grafo sin querer. La copia protege la estructura interna — el caller puede hacer lo que quiera con ret sin afectar al grafo.
  • El orden se invierte (la copia queda al revés). No importa — en una lista de adyacencia el orden de los vecinos es irrelevante.

¿Por qué cada línea?

  • ListaAdyacencia : Grafo — hereda de una clase abstracta Grafo. Así podemos tener una interfaz común con MatrizAdyacencia (polimorfismo).
  • NodoLista<Arista>** listaNodos — es un array de punteros. Cada posición apunta al primer nodo de la lista enlazada de ese vértice (o NULL si no tiene vecinos). Doble puntero porque el array guarda punteros.
  • new NodoLista<Arista>*[V + 1] — V+1 posiciones (0 a V) para poder indexar desde 1 si los vértices van de 1 a V.
  • listaNodos[i] = NULL — lista vacía = puntero a NULL.
  • esPonderado, esDirigido — el grafo sabe qué tipo es. Esto le permite validar (el assert) y automatizar (agregar la arista inversa si es no dirigido).
  • assert(esPonderado || peso == 1) — si el grafo no es ponderado, el peso tiene que ser 1 (el default). Si alguien pasa peso = 5 a un grafo no ponderado, el programa se detiene. Esto atrapa bugs temprano.
  • peso = 1 — parámetro con valor por defecto. Si el grafo no es ponderado, no necesitás pasar el peso: aniadirArista(1, 2) usa peso = 1 automáticamente.
  • new NodoLista<Arista>(a, listaNodos[origen]) — inserción al inicio de la lista (O(1)). El nuevo nodo apunta al que antes era primero.
  • if (!esDirigido) — si el grafo es no dirigido, cada arista existe en ambas direcciones. En vez de que el usuario tenga que acordarse de llamar dos veces, el grafo lo hace solo. Menos errores, más limpio.

Código — Matriz de Adyacencia (notación del profesor)

class MatrizAdyacencia : Grafo {
private:
    int** matrizAdy;                 // matriz V×V
    int V;
    int A;
    bool esPonderado;
    bool esDirigido;
    int INF = std::numeric_limits<int>::max();

public:
    MatrizAdyacencia(int _V, bool _esDirigido, bool _esPonderado) {
        V = _V;
        A = 0;
        esDirigido = _esDirigido;
        esPonderado = _esPonderado;
        matrizAdy = new int*[V + 1];
        for (int i = 0; i <= V; i++) {
            matrizAdy[i] = new int[V + 1];
            for (int j = 0; j <= V; j++) {
                matrizAdy[i][j] = INF;  // INF = no hay arista
            }
        }
    }

    void aniadirArista(int origen, int destino, int peso = 1) {
        assert(esPonderado || peso == 1);
        matrizAdy[origen][destino] = peso;
        A++;
        if (!esDirigido) {
            matrizAdy[destino][origen] = peso;
        }
    }

    NodoLista<Arista>* adyacentesA(int origen) {
        NodoLista<Arista>* ret = NULL;
        for (int i = 1; i <= v; i++) {
            if (matrizAdy[origen][i] != INF) {
                Arista a(origen, i, matrizAdy[origen][i]);
                ret = new NodoLista<Arista>(a, ret);
            }
        }
        return ret;
    }
};

adyacentesA — Lista vs Matriz

Ambas clases tienen la misma firma: NodoLista<Arista>* adyacentesA(int origen). Devuelven lo mismo (una lista de aristas). Pero por dentro hacen cosas muy diferentes:

ListaAdyacencia::adyacentesA(origen) → Recorre la lista enlazada de listaNodos[origen] → Solo mira los vecinos que existen → O(grado(origen)) — proporcional a la cantidad de vecinos MatrizAdyacencia::adyacentesA(origen) → Recorre TODA la fila matrizAdy[origen][1..V] → Chequea cada celda: ¿es != INF? → O(V) — siempre recorre V posiciones, tenga 2 o 2000 vecinos

Esta es exactamente la razón por la que BFS/DFS son O(V+A) con lista pero O(V²) con matriz. Cada llamada a adyacentesA en la matriz cuesta O(V) sin importar cuántos vecinos tenga el nodo. Multiplicado por V nodos = O(V²).

💡 Polimorfismo — por qué importa que la firma sea igual

Como ambas clases heredan de Grafo y tienen la misma firma de adyacentesA, los algoritmos (BFS, DFS, Dijkstra, etc.) se escriben una sola vez usando un puntero Grafo*. El algoritmo llama g->adyacentesA(v) sin saber si por dentro es lista o matriz. Eso es polimorfismo — la misma interfaz, dos implementaciones distintas, y el algoritmo no le importa cuál es.

¿Por qué O(V+A) es tan importante?

🚨 O(V+A) es LA complejidad fundamental de grafos

BFS, DFS, orden topológico — todos son O(V+A). Entender de dónde sale este número es clave para el parcial y para justificar complejidades.

¿De dónde sale O(V+A)?

Pensemos en qué hace un BFS o DFS cuando recorre un grafo con lista de adyacencia:

  • Visitar cada vértice una vez — hay V vértices, así que eso es O(V). Cada vértice se encola/desencola (BFS) o se llama recursivamente (DFS) exactamente una vez gracias al array de visitados.
  • Para cada vértice, recorrer su lista de adyacentes — acá viene lo clave. Cuando procesamos el vértice u, iteramos su lista de vecinos. ¿Cuántos vecinos tiene? Depende del nodo — puede ser 1, puede ser 100. Pero si sumamos la longitud de TODAS las listas de adyacencia, ¿cuánto da?

La suma de todas las listas = A (o 2A)

Grafo no dirigido con arista 1—2: Lista de 1: [2] ← la arista aparece acá Lista de 2: [1] ← y también acá Cada arista aparece en 2 listas → suma total = 2A Grafo dirigido con arista 1→2: Lista de 1: [2] ← la arista aparece solo acá Lista de 2: [] Cada arista aparece en 1 lista → suma total = A

Entonces, cuando BFS/DFS recorre todos los vértices y para cada uno mira todos sus vecinos, el trabajo total es:

Trabajo total = V (visitar cada nodo) + suma de todas las listas = V + A = O(V+A)

No es V × A. No es V². Es V más A — porque cada arista se mira exactamente una o dos veces en total, no una vez por cada vértice.

¿Y si usáramos matriz de adyacencia?

Con una matriz, para saber los vecinos de un nodo tenés que recorrer toda su fila — V posiciones, sin importar cuántos vecinos tenga realmente.

Con lista de adyacencia: Nodo con 3 vecinos → revisamos 3 elementos ✓ Con matriz de adyacencia: Nodo con 3 vecinos → revisamos V elementos (toda la fila) Los otros V-3 son INF o 0 y los miramos al pedo

Resultado: con matriz, BFS/DFS sería O(V²) porque para cada uno de los V nodos recorremos V posiciones. En un grafo disperso (donde A << V²), estamos haciendo mucho trabajo innecesario.

Por eso la lista de adyacencia es casi siempre mejor — iguala a la matriz en el peor caso y la destroza en grafos dispersos (que son la mayoría en la práctica).

⚠️ O(V+A) NO es lo mismo que O(V×A)

O(V+A) = lineal en el tamaño del grafo. Recorrés todo una vez.
O(V×A) = para cada vértice mirás todas las aristas. Mucho más lento (ej: Bellman-Ford).

Ejemplo: V=1000, A=5000.
O(V+A) = 6.000 operaciones.
O(V×A) = 5.000.000 operaciones.

Sección 3 — Implementaciones PARCIAL
Lista de adyacencia: array de listas. Espacio O(V+A). Buena para dispersos. Matriz de adyacencia: matriz V×V. Espacio O(V²). Buena para densos. Acceso O(1).
O(V+A) — de dónde sale: V por visitar cada nodo una vez + la suma de todas las listas de adyacencia = A (o 2A en no dirigido). Cada arista se mira una cantidad fija de veces. ¿Por qué no O(V²)? Porque con lista de adyacencia solo mirás los vecinos reales de cada nodo, no toda una fila de V posiciones. Esta complejidad aplica a: BFS, DFS, orden topológico — todos O(V+A). Es el costo de "tocar todo el grafo una vez".
Pregunta típica: "45 vértices y 1891 aristas, ¿denso o disperso?" → Máximo = 45×44/2 = 990. ¡Tiene más aristas que el máximo! → Es un grafo completo (o con multiaristas).

4 BFS — Búsqueda en Anchura

Explora un grafo visitando todos los vecinos de un nodo antes de pasar al siguiente nivel. Usa una cola (FIFO).

📐 Idea

Empezar en un nodo, visitar todos sus vecinos, después los vecinos de los vecinos, y así sucesivamente — como ondas en el agua.

Pseudocódigo

BFS(start, graph):
  crear cola Q
  marcar start como encolado
  encolar start en Q

  mientras Q no esté vacía:
    desencolar nodo N de Q
    procesar N

    para cada vecino de N:
      si vecino no fue encolado:
        marcar vecino como encolado
        encolar vecino en Q
🚨 ¿Por qué "encolado" y no "visitado"?

El array se marca cuando el nodo entra a la cola, no cuando se procesa. ¿Por qué importa? Porque un nodo puede ser vecino de muchos nodos ya procesados. Si esperás a marcarlo como "visitado" al procesarlo, varios nodos podrían encolarlo al mismo tiempo — queda duplicado en la cola y se procesa varias veces.

Marcándolo como encolado = true en el momento que entra a la cola, garantizás que nunca entra dos veces. Por eso el profesor insiste en llamarlo encolado, no visitado — describe exactamente cuándo se marca.

En DFS sí se llama visitado porque el marcado ocurre al entrar a la función recursiva (al "visitar" el nodo), no al encolar.

Código C++ (notación del profesor)

void BFS(Grafo* grafito, int origen) {
    int v = grafito->getV();
    Queue<int>* cola = new QueueImp<int>();
    bool* encolados = new bool[v + 1];
    for (int i = 0; i <= v; i++)
        encolados[i] = false;

    cola->enqueue(origen);
    encolados[origen] = true;

    while (!cola->isEmpty()) {
        int elemento = cola->dequeue();
        // Procesar elemento

        NodoLista<Arista>* adyacentes = grafito->adyacentesA(elemento);
        while (adyacentes != NULL) {
            Arista arista = adyacentes->dato;
            if (!encolados[arista.destino]) {
                encolados[arista.destino] = true;
                cola->enqueue(arista.destino);
            }
            adyacentes = adyacentes->siguiente;
        }
    }

    delete[] encolados;
    delete cola;
}

¿Por qué cada línea?

  • Grafo* grafito — recibe un puntero a Grafo (la clase abstracta). No importa si por dentro es ListaAdyacencia o MatrizAdyacencia — el polimorfismo se encarga. El BFS funciona igual con cualquier implementación.
  • Queue<int>* cola — cola FIFO de enteros (los vértices). No necesitamos guardar más que el número del vértice en la cola.
  • bool* encolados — array de V+1 posiciones. encolados[i] = true significa que el nodo i ya fue encolado (ya entró a la cola en algún momento). Se marca antes de encolar, no al procesar.
  • grafito->getV() + 1 — +1 para poder indexar desde 1 (vértices de 1 a V).
  • int elemento = cola->dequeue() — sacamos el próximo nodo de la cola. Este es el nodo que vamos a procesar ahora.
  • grafito->adyacentesA(elemento) — pedimos la lista de vecinos. Gracias al polimorfismo, si el grafo es una lista de adyacencia devuelve una copia de la lista enlazada; si es matriz, la construye recorriendo la fila.
  • while (adyacentes != NULL) — recorremos la lista enlazada de aristas. Cada nodo tiene dato (la Arista) y siguiente (puntero al próximo).
  • arista.destino — el vecino al que apunta esta arista.
  • if (!encolados[arista.destino]) — si este vecino nunca fue encolado, lo encolamos y lo marcamos. Si ya fue encolado, lo saltamos — ya va a ser (o ya fue) procesado.

Aplicaciones del BFS — con código

El BFS base es siempre el mismo. Lo que cambia es qué hacés con el array de encolados después (o durante) el recorrido.

¿Es conexo? (grafo no dirigido)

Un BFS desde cualquier nodo. Si todos quedaron encolados → conexo.

bool esConexo(Grafo* grafito) {
    int v = grafito->getV();
    bool* encolados = new bool[v + 1];
    for (int i = 0; i <= v; i++) encolados[i] = false;

    // BFS desde el nodo 1
    Queue<int>* cola = new QueueImp<int>();
    cola->enqueue(1);
    encolados[1] = true;

    while (!cola->isEmpty()) {
        int elemento = cola->dequeue();
        NodoLista<Arista>* ady = grafito->adyacentesA(elemento);
        while (ady != NULL) {
            if (!encolados[ady->dato.destino]) {
                encolados[ady->dato.destino] = true;
                cola->enqueue(ady->dato.destino);
            }
            ady = ady->siguiente;
        }
    }

    // Verificar: ¿todos fueron encolados?
    bool conexo = true;
    for (int i = 1; i <= v; i++) {
        if (!encolados[i]) { conexo = false; break; }
    }
    delete[] encolados;
    delete cola;
    return conexo;
}

Complejidad: O(V+A) — un BFS + un recorrido del array.

Contar componentes conexas (grafo no dirigido)

El array encolados es global (no se reinicia). Cada vez que encontramos un nodo no encolado, es una componente nueva.

int cantComponentes(Grafo* grafito) {
    int v = grafito->getV();
    bool* encolados = new bool[v + 1];
    for (int i = 0; i <= v; i++) encolados[i] = false;

    int componentes = 0;

    for (int i = 1; i <= v; i++) {
        if (!encolados[i]) {
            componentes++;
            // BFS desde i — marca toda su componente
            Queue<int>* cola = new QueueImp<int>();
            cola->enqueue(i);
            encolados[i] = true;
            while (!cola->isEmpty()) {
                int elem = cola->dequeue();
                NodoLista<Arista>* ady = grafito->adyacentesA(elem);
                while (ady != NULL) {
                    if (!encolados[ady->dato.destino]) {
                        encolados[ady->dato.destino] = true;
                        cola->enqueue(ady->dato.destino);
                    }
                    ady = ady->siguiente;
                }
            }
            delete cola;
        }
    }
    delete[] encolados;
    return componentes;
}

Complejidad: O(V+A) total — aunque hay varios BFS, cada nodo y arista se procesa una sola vez (encolados es global).

Contar componentes conexas — versión DFS (recomendada para parcial)

Mismo resultado, mucho menos código. Notar: el array se llama visitado (no encolado).

void DFS_rec(Grafo* grafito, int origen, bool* visitado) {
    visitado[origen] = true;
    NodoLista<Arista>* ady = grafito->adyacentesA(origen);
    while (ady != NULL) {
        if (!visitado[ady->dato.destino])
            DFS_rec(grafito, ady->dato.destino, visitado);
        ady = ady->siguiente;
    }
}

int cantComponentes(Grafo* grafito) {
    int v = grafito->getV();
    bool* visitado = new bool[v + 1];
    for (int i = 0; i <= v; i++) visitado[i] = false;

    int componentes = 0;
    for (int i = 1; i <= v; i++) {
        if (!visitado[i]) {
            DFS_rec(grafito, i, visitado);
            componentes++;
        }
    }
    delete[] visitado;
    return componentes;
}

Complejidad: O(V+A) — igual que BFS. Pero el DFS_rec son ~5 líneas y no necesita cola.

Camino más corto (grafo no ponderado)

BFS descubre nodos por nivel. Guardamos el costo en costo[] y de dónde venimos en vengo[].

void caminoMasCorto(Grafo* grafito, int origen) {
    int v = grafito->getV();
    int INF = std::numeric_limits<int>::max();
    bool* encolados = new bool[v + 1];
    int* costo = new int[v + 1];     // costo[i] = pasos mínimos de origen a i
    int* vengo = new int[v + 1];     // vengo[i] = desde qué nodo llegué a i
    for (int i = 0; i <= v; i++) {
        encolados[i] = false;
        costo[i] = INF;            // INF = no alcanzable (todavía)
        vengo[i] = -1;            // -1 = no tiene predecesor
    }

    Queue<int>* cola = new QueueImp<int>();
    cola->enqueue(origen);
    encolados[origen] = true;
    costo[origen] = 0;

    while (!cola->isEmpty()) {
        int elemento = cola->dequeue();
        NodoLista<Arista>* ady = grafito->adyacentesA(elemento);
        while (ady != NULL) {
            if (!encolados[ady->dato.destino]) {
                encolados[ady->dato.destino] = true;
                costo[ady->dato.destino] = costo[elemento] + 1;
                vengo[ady->dato.destino] = elemento;
                cola->enqueue(ady->dato.destino);
            }
            ady = ady->siguiente;
        }
    }

    // costo[v] = pasos mínimos de origen a v (-1 si no alcanzable)
    // vengo[v] = nodo anterior en el camino más corto
    // Reconstruir camino a v: v → vengo[v] → vengo[vengo[v]] → ... → origen

    delete[] encolados;
    delete[] costo;
    delete[] vengo;
    delete cola;
}

costo[] — cuántos pasos mínimos para llegar de origen a cada nodo. BFS lo llena por nivel: primero costo 0, después costo 1, etc. La primera vez que un nodo se encola es la óptima.

vengo[] — "vengo de..." / "llegué a este nodo desde...". Permite reconstruir el camino completo. Para llegar al nodo 7: vengo[7]=3, vengo[3]=1, vengo[1]=origen → camino: origen→1→3→7.

Complejidad: O(V+A). Solo para grafos no ponderados — si las aristas tienen peso, necesitás Dijkstra.

¿Es fuertemente conexo? (grafo dirigido)

BFS desde cada nodo. Si algún BFS no alcanza a todos → no es fuertemente conexo.

bool esFuertementeConexo(Grafo* grafito) {
    int v = grafito->getV();

    for (int origen = 1; origen <= v; origen++) {
        bool* encolados = new bool[v + 1];
        for (int i = 0; i <= v; i++) encolados[i] = false;

        // BFS desde origen
        Queue<int>* cola = new QueueImp<int>();
        cola->enqueue(origen);
        encolados[origen] = true;
        while (!cola->isEmpty()) {
            int elem = cola->dequeue();
            NodoLista<Arista>* ady = grafito->adyacentesA(elem);
            while (ady != NULL) {
                if (!encolados[ady->dato.destino]) {
                    encolados[ady->dato.destino] = true;
                    cola->enqueue(ady->dato.destino);
                }
                ady = ady->siguiente;
            }
        }

        // ¿Todos encolados?
        for (int i = 1; i <= v; i++) {
            if (!encolados[i]) {
                delete[] encolados;
                delete cola;
                return false;
            }
        }
        delete[] encolados;
        delete cola;
    }
    return true;
}

Complejidad: O(V × (V+A)) — V veces un BFS completo. Es más lento, pero es la forma directa.

Componente conexa más grande (grafo no dirigido)

Variante del contar componentes: contamos nodos en cada BFS y nos quedamos con el máximo.

int componenteMasGrande(Grafo* grafito) {
    int v = grafito->getV();
    bool* encolados = new bool[v + 1];
    for (int i = 0; i <= v; i++) encolados[i] = false;

    int maxTamanio = 0;

    for (int i = 1; i <= V; i++) {
        if (!encolados[i]) {
            int tamanio = 0;
            Queue<int>* cola = new QueueImp<int>();
            cola->enqueue(i);
            encolados[i] = true;
            while (!cola->isEmpty()) {
                int elem = cola->dequeue();
                tamanio++;
                NodoLista<Arista>* ady = grafito->adyacentesA(elem);
                while (ady != NULL) {
                    if (!encolados[ady->dato.destino]) {
                        encolados[ady->dato.destino] = true;
                        cola->enqueue(ady->dato.destino);
                    }
                    ady = ady->siguiente;
                }
            }
            if (tamanio > maxTamanio) maxTamanio = tamanio;
            delete cola;
        }
    }
    delete[] encolados;
    return maxTamanio;
}

Complejidad: O(V+A). Es el ejercicio "letraMasPopular" del parcial Oct'23.

Sección 4 — BFS
Estructura: Cola (FIFO). Primero los vecinos, después los vecinos de los vecinos. Complejidad: O(V+A) — cada vértice y arista se visita una vez. Camino más corto: solo para grafos no ponderados (cada arista = peso 1).
El array se llama "encolado", no "visitado": se marca cuando el nodo ENTRA a la cola, no cuando se procesa. Esto evita que un nodo se encole más de una vez. BFS usa "encolado" (marca al encolar). DFS usa "visitado" (marca al entrar a la recursión). Nombres distintos porque el momento del marcado es distinto.

5 DFS — Búsqueda en Profundidad

Explora un grafo yendo lo más profundo posible por cada rama antes de retroceder. Usa recursión (pila implícita).

Pseudocódigo

DFS(start, graph):
  marcar start como visitado
  procesar start

  para cada vecino de start:
    si vecino no está visitado:
      DFS(vecino, graph)    // recursión

Código C++ (notación del profesor)

void DFS(Grafo* grafito, int origen) {
    int v = grafito->getV();
    bool* visitados = new bool[v + 1];
    for (int i = 0; i <= v; i++)
        visitados[i] = false;
    DFS_rec(grafito, origen, visitados);
    delete[] visitados;
}

void DFS_rec(Grafo* grafito, int origen, bool* visitados) {
    visitados[origen] = true;
    // Procesar el nodo origen

    NodoLista<Arista>* adyacentes = grafito->adyacentesA(origen);
    while (adyacentes != NULL) {
        Arista arista = adyacentes->dato;
        if (!visitados[arista.destino]) {
            DFS_rec(grafito, arista.destino, visitados);
        }
        adyacentes = adyacentes->siguiente;
    }
}
💡 Usos de DFS
  • Detectar ciclos.
  • Detectar componentes fuertemente conexas (Kosaraju).
  • Detectar si un grafo no dirigido es conexo.
  • Contar componentes conexas.
🚨 Tip del profesor para el parcial

Si se puede hacer con BFS o DFS, hacerlo con DFS. Siempre.

¿Por qué? Porque DFS es recursivo — son máximo 5 líneas de código para recorrer un grafo. BFS es iterativo: necesitás crear la cola, encolar, el while, desencolar... mucho más código para escribir en un parcial con tiempo limitado.

6 Puntos de Articulación

📐 Definición

Un punto de articulación es un vértice que, al ser eliminado del grafo, aumenta la cantidad de componentes conexas. Es decir, si lo sacás, el grafo se "rompe" en más partes de las que tenía.

Ejemplo visual

Grafo original (1 componente conexa): 1 — 2 — 3 — 4 — 5 | 6 Si sacamos el nodo 3: Parte A: {1, 2} Parte B: {4, 5} Parte C: {6} → de 1 componente pasamos a 3. El nodo 3 es punto de articulación. Si sacamos el nodo 2: Parte A: {1} Parte B: {3, 4, 5, 6} → de 1 a 2 componentes. El nodo 2 también es punto de articulación. Si sacamos el nodo 5: Queda: {1, 2, 3, 4, 6} — sigue siendo 1 componente. → El nodo 5 NO es punto de articulación (es una "hoja").
💡 ¿Para qué sirven?
  • Redes: si un router es punto de articulación y se cae, la red se divide.
  • Infraestructura: puentes que si se rompen, dejan partes de una ciudad incomunicadas.
  • Apareció en parcial: Oct 2023 — "Explique cómo detectar si existe al menos un punto de articulación."

¿Cómo detectarlos?

💡 La idea es simple: reutilizás cantComponentes

Ya sabés contar componentes conexas con DFS. Para saber si un vértice es punto de articulación, solo necesitás una línea extra: antes de contar, marcás ese vértice como visitado[candidato] = true. Eso lo "desaparece" — el DFS nunca lo toca.

Paso a paso: ¿qué pasa cuando hacés visitado[candidato] = true?

Recordá cómo funciona cantComponentes:

for (int i = 1; i <= v; i++) {
    if (!visitado[i]) {       // ← solo entra si NO está visitado
        DFS_rec(grafito, i, visitado);
        componentes++;
    }
}

Si antes de este loop ponés visitado[3] = true:

  • Cuando el for llega a i=3, !visitado[3] es false → lo saltea. Nunca hace DFS desde el nodo 3.
  • Cuando DFS_rec recorre los vecinos de algún nodo y encuentra al nodo 3 como vecino, chequea !visitado[3] → false → no entra. Nunca lo visita.
  • Resultado: el nodo 3 no existe para el algoritmo. Como si lo hubieras borrado del grafo, pero sin tocar el grafo.

Código del profesor (del pizarrón)

Primero modifica cantConexas para que reciba un int ign (el vértice a ignorar):

int cantConexas(Grafo* g, int ign) {
    int v = g->getV();
    bool* vis = new bool[v + 1]();
    int cont = 0;
    vis[ign] = true;  // ← el truco: "desaparece" este nodo

    for (int origen = 1; origen <= v; origen++) {
        if (!vis[origen]) {
            cont++;
            DFS(origen, vis);
        }
    }
    return cont;
}

Después, esPuntoArt simplemente prueba cada vértice como candidato:

void esPuntoArt(Grafo* g) {
    int v = g->getV();
    for (int i = 1; i <= v; i++) {
        if (cantConexas(g, i) > 1) {
            cout << "El vertice " << i << " es punto de articulacion" << endl;
        }
    }
}

¿Por qué lo separó en dos funciones?

  • cantConexas(g, ign) es la función que ya tenías, con un parámetro extra. Si le pasás ign = 0 (o cualquier nodo que no exista), funciona exactamente igual que antes — cuenta componentes sin ignorar a nadie.
  • esPuntoArt(g) solo la llama V veces con distintos valores de ign. Si alguna vez cantConexas devuelve más de 1, ese vértice es punto de articulación.
  • Reutilización total: no escribís el DFS de nuevo. cantConexas llama al mismo DFS de siempre. esPuntoArt llama a cantConexas. Todo se apila.

Miralo así — solo agregás dos cosas

No necesitás aprender nada nuevo. Es el mismo cantComponentes que ya sabés, adentro de un for, con una línea extra.

⚠️ Complejidad

O(V × (V+A)) — para cada uno de los V vértices, hacemos un DFS completo O(V+A). No es óptimo (existe un algoritmo O(V+A) con DFS y números de descubrimiento), pero es simple de escribir en un parcial y el profesor validó este enfoque en clase.

7 Orden Topológico (Kahn)

Un ordenamiento topológico es una secuencia lineal de vértices de un grafo dirigido tal que para toda arista u→v, u aparece antes que v.

💡 ¿Para qué sirve?

Responde a la pregunta: "¿en qué orden hago las cosas si unas dependen de otras?"

  • Materias en la universidad: no podés cursar Algo 2 sin Algo 1. El orden topológico te da un orden válido para cursar todas las materias respetando las previas.
  • Compilar código: A.cpp depende de B.h, B.h depende de C.h. Orden: C.h → B.h → A.cpp.
  • Tareas de un proyecto: no podés pintar sin aplicar base, y no podés aplicar base sin lijar. Orden: lijar → base → pintar.
  • Recetas de cocina: no podés hornear sin mezclar los ingredientes, y no podés mezclar sin haberlos medido.

El grafo representa dependencias: la arista A→B significa "A tiene que ir antes que B". El orden topológico te da una secuencia que respeta TODAS las dependencias.

¿Cómo detectar que un ejercicio pide orden topológico?

Palabras clave en el enunciado que te deben hacer pensar "esto es orden topológico":

Lo que dice el enunciadoPor qué es orden topológico
"dependencias entre tareas"Hay un orden obligatorio — algunas cosas van antes que otras
"prerrequisitos" / "previas"No podés hacer X sin haber hecho Y primero
"orden de ejecución" / "planificar"Buscás una secuencia válida que respete restricciones
"grafo dirigido acíclico" / "DAG"Directamente te dice la estructura — solo falta aplicar Kahn
"secuencia lineal" / "ordenar vértices"Querés los nodos en fila respetando las aristas
"no hay ciclos" + "dirigido"Si te dicen que es dirigido y sin ciclos = DAG = orden topológico aplica

La señal más clara: si el problema habla de "hacer cosas en un orden que respete dependencias" y el grafo es dirigido sin ciclos → orden topológico.

🚨 Solo funciona en DAGs

Solo es posible si el grafo es un DAG (Directed Acyclic Graph). Si el grafo tiene ciclos, no existe orden topológico. ¿Por qué? Porque un ciclo A→B→C→A significa que A tiene que ir antes que B, B antes que C, y C antes que A — imposible.

📐 Grado de entrada (indegree)

Cada nodo tiene un grado de entrada: la cantidad de aristas que apuntan hacia él. Un nodo con grado 0 significa que nadie depende de él — puede ir primero.

La metáfora de la "bolsa" (el profesor)

Pensá en una bolsa donde vas metiendo nodos que están "listos" (su grado de entrada llegó a 0):

  1. Calcular el grado de entrada de cada nodo (cuántas aristas le llegan).
  2. Meter en la bolsa todos los nodos con grado 0 (no tienen dependencias).
  3. Sacar uno de la bolsa → ese es el siguiente en el orden topológico.
  4. Para cada vecino al que apuntaba el nodo sacado: restarle 1 al grado de entrada (porque ya resolviste esa dependencia).
  5. Si algún vecino llegó a 0 → meterlo en la bolsa (ya no tiene dependencias pendientes).
  6. Repetir desde el paso 3 hasta que la bolsa esté vacía.

Ejemplo paso a paso

Grafo: Tesis → A1, Tesis → A2. A1 → Log, A1 → Rob. A2 → Dis. Rob → Log. Dis → F1, Dis → F2. (Del pizarrón del profe)

VerP1LogP2RobA1CalcF1F2DisA2
Grado entrada010NN0NNNN

Los nodos con grado 0 (P1, P2, Calc, Tesis) van a la bolsa. Sacás uno, le restás 1 a los que apunta. Si alguno llega a 0, entra a la bolsa. Repetís.

💡 ¿Qué es la "bolsa"? Una cola FIFO o una pila

La bolsa puede ser una cola FIFO o una pila — ambas dan un orden topológico válido (pueden ser distintos, pero ambos son correctos). Lo importante es que sacar un elemento sea O(1). Por eso el profe dice que necesitás una estructura con acceso O(1): cola o pila.

Esta es la razón por la que dos implementaciones de Kahn pueden dar salidas diferentes (pregunta de parcial): si usás cola da un orden, si usás pila da otro. Ambos correctos.

Código del profesor

void ordenTopologico(Grafo* g) {
    int V = g->getV();
    int* gradoEntrada = new int[V + 1]();
    Queue<int>* bolsa = new QueueImp<int>();

    // Paso 1: calcular grado de entrada de cada vértice
    for (int i = 1; i <= V; i++) {
        // recorro todas las aristas de i
        NodoLista<Arista>* ady = g->adyacentesA(i);
        while (ady != NULL) {
            int destino = ady->elemento.destino;
            gradoEntrada[destino]++;
            ady = ady->sig;
        }
    }

    // Paso 2: meter en la bolsa los que tienen grado 0
    for (int i = 1; i <= V; i++) {
        if (gradoEntrada[i] == 0)
            bolsa->enqueue(i);
    }

    // Paso 3: procesar la bolsa
    while (!bolsa->isEmpty()) {
        int v = bolsa->dequeue();
        // procesar v (ej: imprimirlo)
        cout << v << " ";

        // restarle 1 al grado de entrada de sus vecinos
        NodoLista<Arista>* ady = g->adyacentesA(v);
        while (ady != NULL) {
            int destino = ady->elemento.destino;
            gradoEntrada[destino]--;
            if (gradoEntrada[destino] == 0)
                bolsa->enqueue(destino);
            ady = ady->sig;
        }
    }

    delete[] gradoEntrada;
    delete bolsa;
}

¿Por qué cada línea?

  • gradoEntrada = new int[V+1]() — el () al final inicializa todo en 0. Cada posición va a contar cuántas aristas apuntan a ese nodo.
  • bolsa = new QueueImp<int>() — la "bolsa" del profesor. Es una cola FIFO, pero podría ser una pila — ambas dan un orden topológico válido.
  • Paso 1 — calcular grados: para cada vértice i, recorro sus aristas. Cada arista i→destino le suma 1 al grado de entrada de destino. Al final, gradoEntrada[j] = cuántas aristas apuntan a j.
  • ady->elemento.destino — el profe usa elemento para acceder al dato del nodo de lista, y sig para el siguiente.
  • Paso 2 — llenar la bolsa: los nodos con grado 0 no tienen dependencias → pueden ir primero. Los metemos en la bolsa.
  • Paso 3 — procesar: sacamos uno de la bolsa, lo procesamos, y recorremos sus vecinos.

El corazón del algoritmo: gradoEntrada[destino]-- y el if

Cuando procesás el nodo v y recorrés sus vecinos, para cada vecino destino hacés:

gradoEntrada[destino]--;
if (gradoEntrada[destino] == 0)
    bolsa->enqueue(destino);

¿Qué significa gradoEntrada[destino]--?

El grado de entrada cuenta cuántos nodos "apuntan a" destino — es decir, cuántas dependencias tiene. Cuando procesás v, estás diciendo "v ya está resuelto". Entonces la arista v→destino ya no cuenta como dependencia pendiente. Le restás 1.

¿Por qué encolar cuando llega a 0?

gradoEntrada[destino] == 0 significa: todas las dependencias de "destino" ya fueron procesadas. Ya no hay ningún nodo que tenga que ir antes que él. Está "libre" — puede entrar a la bolsa y ser procesado.

Ejemplo: nodo 5 tiene gradoEntrada = 3 (nodos 1, 2 y 7 apuntan a 5) Procesamos nodo 1: gradoEntrada[5]-- → queda 2 (falta 2 y 7) Procesamos nodo 7: gradoEntrada[5]-- → queda 1 (falta 2) Procesamos nodo 2: gradoEntrada[5]-- → queda 0 → ENTRA A LA BOLSA Recién cuando TODAS sus dependencias están resueltas, el nodo 5 está listo para ser procesado.

Es como una materia en la universidad: no la podés cursar hasta que aprobaste todas las previas. Cada vez que aprobás una previa, le restás 1. Cuando llega a 0, te podés inscribir.

🚨 Diferencia clave con BFS/DFS/Dijkstra

No hay nodo origen. En BFS/DFS/Dijkstra arrancás desde un nodo específico. En orden topológico arrancás desde TODOS los nodos que tienen grado de entrada 0 — pueden ser varios al mismo tiempo en la bolsa. No elegís uno — metés todos.

💡 Para recordar

Orden topológico = la lista de materias. No podés cursar Algo 2 sin Algo 1. Cuando aprobás una materia (la sacás de la bolsa), les restás 1 a las que dependían de ella. Cuando una materia queda con 0 previas pendientes, la podés cursar (entra a la bolsa).

Sección 6 — Orden Topológico PARCIAL
Solo para DAGs: grafo dirigido sin ciclos. Indegree: cantidad de aristas entrantes a un nodo. Kahn: encolar nodos con indegree 0 → procesar → decrementar vecinos → encolar los que llegan a 0. Complejidad: O(V+A). Pregunta de parcial: "¿Qué hace que dos implementaciones de Kahn den salidas diferentes?" → el orden de desempate al elegir de la cola cuando hay varios nodos con indegree 0.

8 Dijkstra — Camino más corto

Encuentra el camino más corto desde un vértice a todos los demás en un grafo ponderado sin pesos negativos.

🚨 No funciona con pesos negativos

Dijkstra asume que una vez que un nodo es visitado, su distancia es definitiva. Un peso negativo podría mejorar esa distancia después — rompiendo la lógica del algoritmo.

📐 La pregunta clave de Dijkstra

¿Cuál es el vértice no visitado de menor costo?

Dijkstra repite esta pregunta en cada paso. Selecciona ese vértice, lo marca como visitado, y actualiza los costos de sus vecinos. Después vuelve a preguntar.

El algoritmo paso a paso — con la tabla del profesor

Dijkstra mantiene una tabla con 4 columnas para cada vértice:

VerCosto/PesoVisitadoVengo

...INFfalse-
  • Costo: el menor costo conocido para llegar desde el origen a este nodo. Arranca en INF (no conocemos camino) excepto el origen que es 0.
  • Visitado: cuando un nodo se marca como visitado, su costo es definitivo — no va a mejorar.
  • Vengo: desde qué nodo llegué con ese costo. Para reconstruir el camino.

Cada paso de Dijkstra

  1. ¿Cuál es el vértice no visitado de menor costo? → seleccionalo.
  2. Marcalo como visitado → su costo ya es definitivo.
  3. Para cada vecino no visitado: calculá costo[actual] + peso_arista. Si es menor que costo[vecino], actualizá costo[vecino] y vengo[vecino].
  4. Volvé al paso 1.

Terminás cuando todos están visitados o cuando el menor costo entre los no visitados es INF (los que quedan son inalcanzables).

💡 ¿Cómo responder "cuál es el no visitado de menor costo" rápido?

Sin heap (versión simple): recorrés todo el array de costos buscando el mínimo entre los no visitados → O(V) por paso → O(V²) total.

Con min-heap: el tope del heap te da el de menor costo en O(1), sacarlo cuesta O(log V) → O((V+A) log V) total. Mucho mejor para grafos dispersos.

Código del profesor (con Heap)

void dijkstra(Grafo* g, int origen) {
    NodoHeap nh(origen, 0);
    heap->insertar(nh);

    while (!heap->estaVacio()) {
        NodoHeap aProcesar = heap->tope();
        int v = aProcesar.vertice;
        heap->eliminarTope();
        if (visitado[v]) continue;
        visitado[v] = true;

        // recorrer todos los adyacentes
        NodoLista<Arista>* ady = g->adyacentesA(v);
        while (ady != NULL) {
            // w = el destino de la arista
            int w = ady->elemento.destino;
            // costoArista = el costo de dicha arista que recorremos
            int costoArista = ady->elemento.peso;
            // si mejoramos el costo acumulado de llegar a w
            if (costo[w] > costo[v] + costoArista) {
                // actualizamos el costo
                costo[w] = costo[v] + costoArista;
                // actualizamos el vengo
                vengo[w] = v;
                // metemos w en el heap con el nuevo costo
                heap->insertar(NodoHeap(w, costo[w]));
            }
            ady = ady->siguiente;
        }
    }
}

¿Por qué cada línea? (notación del profesor)

  • NodoHeap nh(origen, 0) — crea un nodo para el heap con el vértice origen y costo 0. NodoHeap tiene dos campos: vertice y el costo.
  • heap->tope() — mira el tope del min-heap (el de menor costo) sin sacarlo.
  • heap->eliminarTope() — ahora sí lo saca. Son dos operaciones separadas (peek + remove).
  • int v = aProcesar.vertice — sacamos el número del vértice a procesar.
  • if (visitado[v]) continue — si es una copia vieja (heap lazy), la saltea.
  • int w = ady->elemento.destino — el profe usa elemento para acceder al dato del nodo de la lista (en vez de dato).
  • int costoArista = ady->elemento.peso — lo guarda en una variable con nombre claro en vez de usar ady->elemento.peso inline.
  • costo[w] > costo[v] + costoArista — ¿el camino pasando por v es más barato que el mejor camino conocido a w? Notar: no chequea !visitado[w] porque no hace falta — si w ya está visitado, su costo es definitivo y nunca va a ser mayor que el nuevo, así que el if da false naturalmente.
💡 NodoHeap — qué es

Es un struct simple que el heap usa para ordenar. El heap compara por costo para mantener el de menor costo arriba:

struct NodoHeap {
    int vertice;
    int costo;
    NodoHeap(int v, int c) : vertice(v), costo(c) {}
};

¿Por qué el heap? ¿Por qué sacamos nodos? ¿Por qué "visitado"?

Primero: ¿qué es el heap en Dijkstra?

Pensalo así: el heap es una lista de candidatos — nodos a los que encontraste algún camino pero todavía no estás seguro de que sea el mejor.

Heap = "nodos que tengo pendientes de procesar, ordenados por costo (el más barato arriba)" Al inicio: solo el origen (costo 0). A medida que procesás nodos, descubrís caminos a nuevos vecinos → los metés al heap con su costo.

¿Por qué sacamos el de arriba?

Dijkstra en cada paso pregunta: "¿cuál es el candidato más barato?". El heap te lo responde en O(1) — es el que está arriba (el tope). Lo sacás porque vas a procesarlo ahora.

Sin heap, tendrías que recorrer todo el array de costos buscando el mínimo → O(V) cada vez. Con heap → O(log V). Esa es la única razón del heap: contestar "¿quién es el más barato?" rápido.

¿Por qué marcamos como visitado? ¿Por qué no volver a mirarlo?

Cuando sacás el nodo más barato del heap, su costo es definitivo. ¿Por qué? Pensalo:

  • Este nodo tiene el menor costo de todos los que quedan en el heap.
  • Todos los caminos futuros a este nodo pasarían por algún otro nodo que todavía está en el heap.
  • Pero esos otros nodos ya tienen costo mayor o igual (si no, habrían salido antes).
  • Y los pesos son no negativos, así que sumarle más aristas solo puede aumentar el costo.
  • Conclusión: no existe un camino más barato a este nodo. Su costo es final.

Por eso lo marcamos como visitado = true — "ya terminé con este nodo, su costo no va a cambiar nunca".

🚨 Esto es EXACTAMENTE por qué Dijkstra no funciona con pesos negativos

Si una arista pudiera tener peso negativo, un nodo que todavía está en el heap podría ofrecer un camino más barato a un nodo que ya visitaste — porque sumarle un peso negativo lo haría más barato. La garantía de "si salió del heap, su costo es final" se rompe. Por eso necesitás Bellman-Ford para pesos negativos.

El problema del heap: nodos duplicados

⚠️ ¿Qué pasa cuando encontrás un camino mejor a un nodo que ya está en el heap?

Ejemplo: el nodo 5 ya está en el heap con costo 20. Después descubrís que podés llegar con costo 12. ¿Qué hacés? El nodo 5 ya está adentro del heap con el costo viejo. Hay dos formas de resolverlo:

Opción 1: Heap "lazy" — insertar duplicados y filtrar

La más simple. Cuando encontrás un costo mejor, insertás el nodo de nuevo en el heap con el nuevo costo. No borrás el viejo. El heap puede tener el mismo nodo varias veces con distintos costos.

  • Cuando sacás un elemento del heap, chequeás: ¿ya está visitado?
  • Si sí → es una copia vieja con costo desactualizado → lo ignorás (continue) y sacás el siguiente.
  • Si no → es la versión con el mejor costo → lo procesás.
while (!heap.estaVacio()) {
    VertexDistance actual = heap.top(); heap.pop();

    if (visitado[actual.vertex]) continue;  // ← copia vieja, ignorar
    visitado[actual.vertex] = true;

    // procesar vecinos...
    // si encontrás mejor costo → insertás de nuevo en el heap
    heap.insert(VertexDistance(vecino, nuevoCosto));  // duplicado, no importa
}

Ventaja: simple de implementar. No necesitás buscar ni modificar elementos dentro del heap.

Desventaja: el heap puede crecer hasta tamaño A (una entrada por cada arista que mejora un costo) en vez de V. En el peor caso V² entradas. Pero funciona correctamente — las copias viejas se descartan con el if (visitado) continue.

Complejidad: O((V+A) log A) en vez de O((V+A) log V). En la práctica, la diferencia es mínima porque log A ≈ log V² = 2 log V.

Opción 2: Cola de Prioridad Extendida (Heap + Hash)

La más eficiente. Usás el Heap + Hash que viste en el tema de Cola de Prioridad Extendida. Cuando encontrás un costo mejor:

  • La hash te dice dónde está ese nodo dentro del heap → O(1).
  • Le cambiás la prioridad (el costo) directamente → O(1).
  • Después hacés flotar (porque el costo bajó, el nodo tiene que subir en el min-heap) → O(log V).

El heap siempre tiene exactamente V elementos (uno por nodo, sin duplicados). Cada nodo aparece una sola vez.

Ventaja: heap más chico (tamaño V), complejidad limpia O((V+A) log V).

Desventaja: tenés que implementar la Cola de Prioridad Extendida (heap con hash que trackea posiciones + changePriority). Más complejo de escribir.

9 Bellman-Ford

Camino más corto desde un vértice a todos los demás. Sí funciona con pesos negativos (a diferencia de Dijkstra).

📐 Idea general

Es como Dijkstra pero sin la garantía de "visitado = definitivo". Como los pesos pueden ser negativos, un nodo puede mejorar su costo varias veces. Entonces:

  1. Inicializar: costo[origen] = 0, resto = INF.
  2. Usar una cola FIFO (no un heap) con el nodo origen.
  3. Para cada nodo sacado de la cola: recorrer vecinos, si mejora → actualizar y re-encolar.
  4. Un nodo puede entrar y salir de la cola varias veces.

Código (notación del profesor)

void bellmanFord(Grafo* g, int origen) {
    int v = g->getV();
    int* costo = new int[v + 1];
    int* vengo = new int[v + 1];
    bool* encolado = new bool[v + 1];
    for (int i = 0; i <= v; i++) {
        costo[i] = INF; vengo[i] = -1; encolado[i] = false;
    }
    costo[origen] = 0;

    Queue<int>* cola = new QueueImp<int>();
    cola->enqueue(origen);
    encolado[origen] = true;

    while (!cola->isEmpty()) {
        int u = cola->dequeue();
        encolado[u] = false;  // puede volver a entrar

        NodoLista<Arista>* ady = g->adyacentesA(u);
        while (ady != NULL) {
            int w = ady->elemento.destino;
            int costoArista = ady->elemento.peso;
            if (costo[u] + costoArista < costo[w]) {
                costo[w] = costo[u] + costoArista;
                vengo[w] = u;
                if (!encolado[w]) {
                    cola->enqueue(w);
                    encolado[w] = true;
                }
            }
            ady = ady->sig;
        }
    }
    // costo[i] = costo mínimo de origen a i
    // vengo[i] = desde qué nodo llegué a i
    delete[] costo; delete[] vengo; delete[] encolado;
    delete cola;
}

¿Por qué encolado y no visitado?

En Bellman-Ford usamos encolado (como en BFS), no visitado (como en Dijkstra). ¿Por qué?

  • En Dijkstra, visitado = true significa "terminé, nunca más lo toco".
  • En Bellman-Ford, un nodo puede mejorar después de ser procesado (por un peso negativo). Entonces no hay "visitado definitivo".
  • encolado solo significa "ya está en la cola, no lo metas de nuevo". Pero cuando sale (encolado = false), puede volver a entrar si alguien le mejora el costo.
💡 Para recordar

Bellman-Ford es el Dijkstra pesimista. Dijkstra confía: "si ya lo visité, no va a mejorar". Bellman-Ford no confía en nadie: "puede que mejore, dejalo volver a la cola". Por eso acepta negativos (donde Dijkstra se rompe) pero es más lento — O(V×A) vs O((V+A)logV).

Array: encolado (no visitado) — porque un nodo puede volver a entrar. Complejidad: O(V × A) — más lento que Dijkstra. Cuándo usarlo: cuando hay aristas con peso negativo. Si no hay negativos, Dijkstra es mejor.

10 Floyd-Warshall

Camino más corto entre todos los pares de vértices. Simple de implementar. Acepta pesos negativos (no ciclos negativos).

📐 La idea: clausura transitiva

El profesor habla de transitiva — es la clausura transitiva de los caminos:

  • Si hay camino de A a K, y camino de K a B, entonces hay camino de A a B pasando por K.
  • Floyd pregunta para CADA nodo K: "¿puedo mejorar el camino de i a j pasando por K?"
  • Si costo[i][k] + costo[k][j] < costo[i][j] → sí, encontré un camino mejor pasando por K.

Al probar TODOS los nodos como intermediarios (k = 1, 2, ..., V), al final tenés el camino más corto entre todos los pares.

Matriz de Clausura Transitiva

El profesor la llama así: la matriz de clausura transitiva. ¿Qué significa?

La transitividad dice: si puedo ir de A a B, y de B a C, entonces puedo ir de A a C. La clausura transitiva es aplicar esa regla hasta agotar todas las posibilidades.

Antes de Floyd (solo aristas directas): 1→2 (costo 5), 2→3 (costo 3) ¿1→3? No hay arista directa → costo[1][3] = INF Después de Floyd (clausura transitiva): 1→3 pasando por 2: costo[1][2] + costo[2][3] = 5 + 3 = 8 costo[1][3] = 8 ← encontró el camino transitivo

Floyd cierra la matriz — al terminar, si existe un camino de i a j (por cualquier cantidad de nodos intermedios), costo[i][j] tiene el costo mínimo. Si no existe camino, queda en INF.

💡 Versión simplificada: solo saber si hay camino (sin costos)

Si solo te interesa saber si existe camino de i a j (sí/no, sin importar el costo), la clausura transitiva se puede hacer con una matriz de bools:

// alcanzable[i][j] = true si hay algún camino de i a j
for (int k = 1; k <= v; k++)
    for (int i = 1; i <= v; i++)
        for (int j = 1; j <= v; j++)
            if (alcanzable[i][k] && alcanzable[k][j])
                alcanzable[i][j] = true;

Mismo triple loop, pero en vez de sumar costos y comparar, solo propagás "si puedo llegar a k y desde k puedo llegar a j, entonces puedo llegar de i a j". Sirve para determinar conexidad en grafos dirigidos.

Paso a paso — ¿qué hace el loop de K?

Imaginá que tenés 4 nodos. Floyd prueba:

k=1: Para cada par (i,j): ¿conviene ir de i a j pasando por el nodo 1? costo[i][1] + costo[1][j] < costo[i][j]? → si sí, actualizar k=2: Para cada par (i,j): ¿conviene ir de i a j pasando por el nodo 2? (ya considerando los caminos mejorados por k=1) k=3: Para cada par (i,j): ¿conviene pasando por 3? (ya considerando mejoras de k=1 y k=2) k=4: Para cada par (i,j): ¿conviene pasando por 4? Al final: costo[i][j] tiene el camino más corto de i a j, usando cualquier combinación de nodos intermedios.

Cada iteración de K "abre" un nodo más como posible intermediario. Al terminar, todos los nodos fueron probados como intermediarios — clausura transitiva completa.

La matriz — ¿qué representa costo[i][j]?

Es una matriz V×V donde cada celda guarda el costo mínimo para ir de i a j:

1 2 3 4 1 [ 0 5 3 7 ] ← costos desde nodo 1 2 [ INF 0 1 5 ] ← costos desde nodo 2 3 [ INF INF 0 4 ] ← costos desde nodo 3 4 [ INF INF INF 0 ] ← costos desde nodo 4 costo[1][3] = 3 → el camino más barato de 1 a 3 cuesta 3 costo[2][4] = 5 → el camino más barato de 2 a 4 cuesta 5 costo[4][1] = INF → no hay camino de 4 a 1

Código (notación del profesor)

void floydWarshall(Grafo* g) {
    int v = g->getV();
    // Inicializar la matriz con las aristas del grafo
    int** costo = new int*[v + 1];
    int** vengo = new int*[v + 1];
    for (int i = 1; i <= v; i++) {
        costo[i] = new int[v + 1];
        vengo[i] = new int[v + 1];
        for (int j = 1; j <= v; j++) {
            costo[i][j] = INF;
            vengo[i][j] = -1;
        }
        costo[i][i] = 0;  // de i a i cuesta 0
    }

    // Cargar las aristas del grafo en la matriz
    for (int i = 1; i <= v; i++) {
        NodoLista<Arista>* ady = g->adyacentesA(i);
        while (ady != NULL) {
            int dest = ady->elemento.destino;
            int peso = ady->elemento.peso;
            costo[i][dest] = peso;
            vengo[i][dest] = i;
            ady = ady->sig;
        }
    }

    // El triple loop — clausura transitiva
    for (int k = 1; k <= v; k++) {
        for (int i = 1; i <= v; i++) {
            for (int j = 1; j <= v; j++) {
                if (costo[i][k] < INF && costo[k][j] < INF
                    && costo[i][k] + costo[k][j] < costo[i][j]) {
                    costo[i][j] = costo[i][k] + costo[k][j];
                    vengo[i][j] = vengo[k][j];
                }
            }
        }
    }

    // costo[i][j] = costo mínimo de i a j
    // vengo[i][j] = nodo anterior a j en el camino de i a j
}

¿Por qué cada línea?

  • costo[i][i] = 0 — ir de un nodo a sí mismo cuesta 0.
  • costo[i][k] < INF && costo[k][j] < INF — verificar que existan caminos de i a k y de k a j. Si alguno es INF (no hay camino), sumarlos daría overflow.
  • costo[i][k] + costo[k][j] < costo[i][j] — ¿es más barato ir de i a j pasando por k que el camino actual?
  • vengo[i][j] = vengo[k][j] — si el camino mejora pasando por k, el nodo anterior a j en el nuevo camino es el mismo que en el camino de k a j.
  • k va PRIMERO (loop externo) — esto es crucial. Si k fuera el loop interno, no funcionaría porque los caminos intermedios no estarían calculados todavía.

11 Árbol de Cubrimiento Mínimo (MST)

Un MST es un subconjunto de aristas que conecta todos los vértices sin ciclos y con el menor costo total. Se aplica a grafos no dirigidos y ponderados.

Prim

📐 Idea
  1. Empezar con un vértice arbitrario.
  2. Seleccionar la arista más liviana que conecte al MST con un nodo que no está en el MST.
  3. Añadir ese nodo al MST.
  4. Repetir hasta incluir todos los vértices.

Kruskal

📐 Idea
  1. Ordenar todas las aristas de menor a mayor peso.
  2. Para cada arista: si no forma ciclo, agregarla al MST.
  3. Repetir hasta tener V−1 aristas.

Para detectar ciclos eficientemente usa el MFSet (Conjunto Disjunto).

MFSet — Conjunto Disjunto

Estructura para mantener conjuntos disjuntos. Dos operaciones: Find (¿a qué conjunto pertenece?) y Merge/Union (unir dos conjuntos).

Código MFSet (con Path Compression + Union by Rank)

class DisjointSet {
private:
    int* parent;
    int* heigth;
    int size;
public:
    DisjointSet(int n) {
        size = n;
        parent = new int[n + 1];
        heigth = new int[n + 1];
        for (int i = 0; i <= n; i++) {
            parent[i] = i;    // cada nodo es su propio padre
            heigth[i] = 0;
        }
    }

    int find(int x) {
        if (parent[x] == x) return x;
        parent[x] = find(parent[x]); // path compression
        return parent[x];
    }

    void merge(int x, int y) {
        int xRoot = find(x), yRoot = find(y);
        if (xRoot == yRoot) return;
        // Union by rank:
        if (heigth[xRoot] < heigth[yRoot])
            parent[xRoot] = yRoot;
        else if (heigth[xRoot] > heigth[yRoot])
            parent[yRoot] = xRoot;
        else {
            parent[yRoot] = xRoot;
            heigth[xRoot]++;
        }
    }
};
💡 Para recordar

Kruskal es el tacaño: ordena todas las aristas por precio y va comprando la más barata que no le haga un loop. Prim es el primo de Dijkstra: crece desde un nodo, siempre elige la conexión más barata al vecindario. Dos personalidades, mismo resultado (el MST).

Sección 11 — MST (Prim & Kruskal) PARCIAL
MST: subconjunto de aristas, sin ciclos, que conecta todos los nodos con costo mínimo. Solo para grafos no dirigidos ponderados.
Prim: crece el MST desde un nodo. Complejidad O((V+A)logV) con heap. Casi igual a Dijkstra (cambia 1 línea). Kruskal: ordena aristas por peso, agrega si no forma ciclo. Usa MFSet. Complejidad O(A log A).
MFSet: Find (¿mismo conjunto?) + Merge (unir conjuntos). Con path compression + union by rank → casi O(1) amortizado. Path compression: en find(), apuntar directamente a la raíz. Union by rank: colgar el árbol más bajo del más alto.

Patrón 1: "Usar X para determinar si es fuertemente conexo"

Con Dijkstra

Dijkstra desde un origen te da el costo mínimo a todos los demás. Si algún nodo queda en INF, no es alcanzable desde ese origen.

¿Es fuertemente conexo? = ¿Desde CUALQUIER nodo puedo llegar a CUALQUIER otro? 1. Correr Dijkstra desde CADA nodo (V veces) 2. Después de cada Dijkstra, verificar: ¿algún costo[i] quedó en INF? → SÍ: no es fuertemente conexo (ese nodo no llega a todos) → NO: ese nodo llega a todos, seguir con el próximo Si los V Dijkstras dieron todos OK → fuertemente conexo

Complejidad: O(V × (V+A) log V). Es más lento que usar BFS/DFS V veces, pero si el profe te dice "usar Dijkstra", es lo que tenés que hacer.

Con BFS/DFS (la forma normal)

Mismo concepto pero más simple y rápido:

1. DFS/BFS desde cada nodo 2. Si en algún recorrido no se visitan todos → no es fuertemente conexo Complejidad: O(V × (V+A))

Patrón 2: "Usar X para contar componentes conexas"

Con DFS (lo más rápido de escribir)

cantConexas(g, -1) // el -1 = no ignorar ningún nodo → visitado[] global, for de 1 a V, cada DFS nuevo = componente++ Complejidad: O(V+A)

Con Dijkstra

Mismo patrón que cantConexas pero usando Dijkstra en vez de DFS: 1. Para cada nodo no procesado, correr Dijkstra 2. Todos los nodos con costo != INF son parte de esa componente 3. Cada Dijkstra nuevo = componente++ Complejidad: O(V × (V+A) log V) — exagerado, pero funciona

¿Cuándo te lo piden?

Para verificar débilmente conexo: un grafo dirigido es débilmente conexo si al ignorar las direcciones queda conexo.

¿Es débilmente conexo? 1. Construir el "grafo subyacente no dirigido": Para cada arista u→v del grafo original, agregar u→v Y v→u en un grafo nuevo 2. Correr DFS/BFS una vez en el grafo nuevo 3. ¿Todos visitados? → débilmente conexo

Patrón 8: "Justificá la complejidad"

No alcanza con decir "O(V+A)" — tenés que explicar de dónde sale

  • "¿Por qué O(V+A)?" → "Porque visitamos cada nodo una vez (V) y recorremos cada arista una vez en la lista de adyacencia (A). Cada nodo y arista se procesa una cantidad constante de veces."
  • "¿Por qué Dijkstra es O((V+A)logV)?" → "Vaciamos el heap: V extracciones × O(log V) = O(V log V). Para cada arista, potencialmente insertamos en el heap: A inserciones × O(log V) = O(A log V). Total: O((V+A) log V)."
  • "¿Por qué Floyd es O(V³)?" → "3 loops anidados de 1 a V. Cada uno itera V veces. V × V × V = V³."
  • "¿Por qué Build-Heap es O(N) y no O(N log N)?" → "Porque los nodos cerca de las hojas (la mayoría) bajan poco. La suma de hundimientos es 1×N/2 + 2×N/4 + 3×N/8 + ... = O(N) por serie geométrica."

Patrón 9: "Siempre hay una trampa en el enunciado"

Cosas que te hacen equivocar si no leés bien

  • "Grafo ponderado y completo, Dijkstra da mal" → tiene pesos negativos (la trampa).
  • "Componentes conexas con al menos 3 vértices" → no contar las de 1 o 2 (la trampa).
  • "Dos implementaciones de Kahn dan distintos resultados" → el orden de desempate cuando hay varios con indegree 0 (cola vs pila, o FIFO vs LIFO).
  • "Obtener mínimo O(1) en AVL" → cachear, pero ¿qué pasa con las demás operaciones? (nada cambia).
  • "Grafo acíclico, no dirigido, no conexo, ¿orden topológico?" → NO, porque necesita ser DIRIGIDO (la trampa).
  • "Hash tabla con problemas de rendimiento" → no es cambiar la función ni agrandar — es rehashing (re-insertar todo).

AVL

P1📋 Oct'23
Además de las rotaciones simples (derecha e izquierda), los árboles AVL utilizan rotaciones dobles. Describí las rotaciones dobles (derecha-izquierda e izquierda-derecha) y cuándo se aplican. ¿Qué orden tienen estas rotaciones frente a las simples?
Respuesta

Doble izquierda-derecha (LR): el desbalance está en el hijo izquierdo del hijo derecho de Z. Se aplica cuando Z tiene bf = -2 y su hijo Y tiene bf = +1. Primero se rota Y a la izquierda, después Z a la derecha.

Doble derecha-izquierda (RL): el desbalance está en el hijo derecho del hijo izquierdo de Z. Se aplica cuando Z tiene bf = +2 y su hijo Y tiene bf = -1. Primero se rota Y a la derecha, después Z a la izquierda.

Orden: O(1) — igual que las simples. Son 2 rotaciones simples, cada una O(1), entonces O(1) total. La diferencia es cuándo se aplican, no su costo.

P2📋 Oct'23
Compará el orden de tiempo de Insertar, Borrar mínimo y Obtener mínimo para una cola de prioridad implementada con Heap y con AVL. Indicá cómo podría lograr que el Obtener mínimo para un AVL tenga tiempo constante, y qué impacto tendría en las demás operaciones.
Respuesta

Insertar: Heap O(log n), AVL O(log n).

Borrar mínimo: Heap O(log n), AVL O(log n).

Obtener mínimo: Heap O(1) (la raíz), AVL O(log n) (hay que ir hasta la hoja más a la izquierda).

¿Cómo hacer getMin O(1) en AVL? Cachear el mínimo en una variable. Actualizarlo en insert (si el nuevo elemento es menor) y en delete (si se borró el mínimo, buscar el nuevo mínimo en O(log n)). Las demás operaciones no cambian de orden — solo el delete del mínimo tiene un costo extra de buscar el nuevo, pero eso ya era O(log n).

P3📋 May'23
Explicá la relación entre "AVL" y "árbol auto-balanceado".
Respuesta

Un AVL es un tipo de árbol auto-balanceado, no el único. "Árbol auto-balanceado" es la categoría general: árboles binarios de búsqueda que mantienen su altura en O(log n) automáticamente después de cada operación. AVL logra esto usando el factor de balance (diferencia de alturas entre subárboles) y rotaciones cuando |bf| > 1. Otros árboles auto-balanceados: Red-Black trees, B-trees.

En una tabla de hash cerrada, ¿por qué debemos marcar los elementos eliminados como "eliminados" y no como NULL?
Respuesta

Porque en hash cerrada, la búsqueda sigue un camino de probing (lineal, cuadrático o doble hashing). Si un elemento en el medio del camino se borra y se pone NULL, la búsqueda de otro elemento que pasó por esa posición se detendría prematuramente pensando que no existe — cuando en realidad está más adelante en el camino.

El tombstone ("eliminado") le dice a la búsqueda: "acá hubo algo, seguí buscando". Así no se rompe el camino de probing.

P5📋 Oct'24
Usted tiene una tabla de hash dando problemas de rendimiento. Su compañero ofrece: A. Cambiar la función de hash dejando la tabla con sus datos inalterada. B. Duplicar el tamaño del array copiando los elementos a las mismas posiciones. Comentá qué problemas podría tener cada solución.
Respuesta

A. Cambiar la función de hash sin mover datos: DESASTRE. Los elementos fueron insertados usando la función vieja — sus posiciones dependen de ella. Con la nueva función, la búsqueda calcula posiciones diferentes y no encuentra los datos. Hay que hacer rehashing: re-insertar todos los elementos con la nueva función.

B. Duplicar tamaño copiando a las mismas posiciones: NO SIRVE. La función de hash usa % tamaño. Si el tamaño cambia, la función da resultados diferentes. Los elementos quedan en posiciones que no corresponden al nuevo tamaño. También requiere rehashing.

La solución correcta: crear una tabla nueva (con mejor función o más grande), re-insertar todos los elementos usando la nueva función. Eso es rehashing.

P6📋 May'24
Explicá las similitudes y diferencias del factor de carga en el hash abierto vs hash cerrado.
Respuesta

Factor de carga α = n/m (elementos / tamaño de tabla). En ambos casos mide "qué tan llena está la tabla".

Hash abierto (chaining): α puede ser > 1 (más elementos que posiciones, las listas crecen). Se recomienda rehashing cuando α > 1.0. El rendimiento degrada linealmente con α (búsqueda promedio = 1 + α/2).

Hash cerrado (direccionamiento abierto): α NUNCA puede ser > 1 (cada posición tiene un solo elemento). Se recomienda rehashing cuando α > 0.7-0.75. El rendimiento degrada mucho más rápido que en abierto cuando α se acerca a 1 (clustering).

P7📋 May'23
¿Para qué utilizaríamos un heap en conjunto con una tabla de hash?
Respuesta

Para implementar una Cola de Prioridad Extendida. El heap mantiene el orden de prioridades (insert/deleteMin en O(log n), getMin en O(1)), pero no puede encontrar un elemento arbitrario eficientemente — buscar x es O(n).

La hash mapea elemento → posición en el heap en O(1). Así, operaciones como changePriority(x) y delete(x) pasan de O(n) a O(log n): la hash encuentra x en O(1), y después swim/sink es O(log n).

P8📋 Oct'24
Tenés un array de enteros que representa un min-heap válido. Devolvé un nuevo array con los mismos datos pero que represente un max-heap. Hacerlo en O(N) promedio. Aclará si usás base 0 o base 1.
Respuesta

La clave: el orden del min-heap no sirve para nada para el max-heap. Simplemente copiá los elementos al nuevo array y hacé Build-Heap (heapify) para max-heap.

Build-Heap: recorrer desde n/2 hasta 1 (o 0 en base 0), llamando hundir(i) para cada posición. Esto construye un heap válido en O(N) — no O(N log N) como insertar de a uno.

¿Por qué O(N)? Porque la mayoría de los nodos están cerca de las hojas y casi no bajan. La suma total de hundimientos es N, no N log N (demostración por serie geométrica).

P9📋 May'23
¿Cómo puedes implementar un heap que permita la eliminación de elementos no solo desde la raíz? (No usar estructuras de datos auxiliares). Describí el algoritmo y su complejidad.
Respuesta

Sin estructuras auxiliares, hay que escanear el array para encontrar el elemento:

  1. Encontrar x escaneando el array → posición i. O(n).
  2. Swap arr[i] con arr[sigLibre-1]. Después sigLibre--.
  3. Llamar flotar(i). Después hundir(i). Siempre los dos, sin pensar. Si tiene que subir, flotar lo mueve y hundir no hace nada. Si tiene que bajar, flotar no hace nada y hundir lo mueve.

Complejidad: O(n) — el scan domina. Con hash sería O(log n).

Grafos — Teoría

P10📋 Oct'24
¿Para solucionar qué tipo de problema y en qué tipo de grafos es más eficiente usar Floyd en lugar de Dijkstra V veces? Justificá brevemente.
Respuesta

Problema: camino más corto entre todos los pares de vértices.

Tipo de grafo: grafos densos (A ≈ V²) o con pesos negativos.

¿Por qué? Floyd = O(V³). Dijkstra V veces = O(V(V+A)logV). En un grafo denso, A ≈ V², entonces Dijkstra V veces ≈ O(V³ log V) — más lento que Floyd. Además, si hay pesos negativos, Dijkstra directamente no funciona.

En grafos dispersos, Dijkstra V veces gana: O(V² log V) vs O(V³).

P11📋 Oct'24
Explicá las diferencias entre los algoritmos de Kruskal y Prim para encontrar el Árbol de Cobertura Mínima.
Respuesta

Kruskal: ordena TODAS las aristas por peso. Recorre de menor a mayor, agregando cada arista al MST si no forma ciclo (usa MFSet para detectar ciclos). Trabaja arista por arista. Complejidad: O(A log A).

Prim: empieza desde un vértice y crece el MST nodo por nodo. En cada paso selecciona la arista más barata que conecta el MST con un nodo que no está en él. Similar a Dijkstra (cambia 1 línea). Complejidad: O((V+A) log V) con heap.

Diferencias: Kruskal ve las aristas globalmente (ordenadas). Prim crece localmente (desde un nodo). Kruskal necesita MFSet. Prim necesita heap. Ambos dan el mismo MST (si los pesos son únicos).

P12📋 May'24
Defina qué es un grafo: i. Conexo, ii. No conexo, iii. Fuertemente conexo, iv. Débilmente conexo.
Respuesta

i. Conexo (no dirigido): existe un camino entre cualquier par de nodos.

ii. No conexo: existen nodos que no están conectados — hay más de una componente conexa.

iii. Fuertemente conexo (dirigido): hay camino dirigido de u a v Y de v a u, para todo par (u,v). Se puede llegar de cualquier nodo a cualquier otro siguiendo las direcciones.

iv. Débilmente conexo (dirigido): si ignoramos las direcciones de las aristas (lo tratamos como no dirigido), el grafo resultante es conexo. Pero siguiendo las direcciones, no necesariamente podés ir de cualquiera a cualquiera.

P13📋 Oct'23
a. ¿Cuál es la diferencia entre un grafo denso y un grafo completo? b. Si debés implementar un grafo, ¿qué estructura utilizarías para las aristas si es denso o disperso? c. Tenés un grafo no dirigido, ponderado y completo. Después de ejecutar Dijkstra desde un origen, la distancia obtenida a un destino X no es correcta. ¿Qué sucedió?
Respuesta

a. Completo = tiene TODAS las aristas posibles (V(V-1)/2 en no dirigido). Denso = tiene MUCHAS aristas pero no necesariamente todas. Todo grafo completo es denso, pero no todo denso es completo.

b. Denso → matriz de adyacencia (O(V²) espacio, pero acceso O(1) a cualquier arista). Disperso → lista de adyacencia (O(V+A) espacio, no desperdicia).

c. Si Dijkstra dio un resultado incorrecto, el grafo tiene aristas con peso negativo. Dijkstra asume que una vez visitado un nodo, su costo es definitivo. Un peso negativo puede mejorar un nodo ya visitado, rompiendo esa garantía.

P14📋 Oct'24
Implementá la función Lista<Arista*>* quieroCobrar(Grafo * g) que utilice un algoritmo de grafos para encontrar los cables con longitud total mínima para conectar todos los escenarios de un festival. El grafo tiene: getVertices(), adyacentes(int v).

P16📋 May'24
Implementá void clasificarGrafo(Grafo * g) que dado un grafo dirigido imprima: "fuertemente conexo", "débilmente conexo", o "no conexo".
Respuesta

1. ¿Fuertemente conexo? DFS desde cada nodo (V veces). Si todos los DFS visitan todos los nodos → fuertemente conexo.

2. ¿Débilmente conexo? Construir el grafo subyacente no dirigido (para cada arista u→v, agregar también v→u). DFS desde cualquier nodo en el grafo no dirigido. Si visita todos → débilmente conexo.

3. Si ni siquiera débilmente: no conexo.

Complejidad: paso 1 = O(V(V+A)), paso 2 = O(V+A). Total O(V(V+A)).

Mismo patrón que cantConexas, pero con tamaño++ en cada nodo visitado y if (tamaño > maxTamaño) maxTamaño = tamaño al terminar cada DFS.

P18📋 May'24
Dado un grafo no dirigido, no ponderado y no conexo, implementá int cantCompConexas(Grafo *g). Justificá su orden.
Respuesta

DFS iterativo con visitado global:

int cantCompConexas(Grafo* g) {
    int v = g->getV();
    bool* visitado = new bool[v+1];
    for (int i = 0; i <= v; i++) visitado[i] = false;
    int comp = 0;
    for (int i = 1; i <= v; i++) {
        if (!visitado[i]) {
            DFS_rec(g, i, visitado);
            comp++;
        }
    }
    delete[] visitado;
    return comp;
}

Orden: O(V+A). Cada nodo se visita una vez (O(V)). Cada arista se recorre una vez por cada extremo en la lista de adyacencia (O(A)). El for externo es O(V) pero el trabajo total de todos los DFS es O(V+A) porque visitado es global.

📐 Problema en una oración

Un investigador recorre n salas en línea. En cada sala recoge una llave y después enfrenta una puerta. ¿Cuántas llaves adicionales necesita pedir antes de empezar?

Paso 1 — Entender la entrada

⚠️ Formato de entrada

Línea 1: un entero n — cantidad de salas (2 ≤ n ≤ 10⁷).

Línea 2: una cadena s de longitud 2n−2.

¿Cómo leer la cadena?

La cadena alterna: llave, puerta, llave, puerta, ... Son n−1 pares (porque hay n−1 puertas entre n salas).

n = 5, s = "xYyXzZaZ" (largo = 2×5−2 = 8)

Posición01234567
CarácterxYyXzZaZ
Tipo llavepuerta llavepuerta llavepuerta llavepuerta
Sala Sala 1 Sala 2 Sala 3 Sala 4

Indices pares (0, 2, 4, 6) = llaves (minúsculas)   |   Indices impares (1, 3, 5, 7) = puertas (mayúsculas)

¿Cómo saber qué llave y qué puerta corresponden a cada sala?

Las salas van de 1 a n−1 (la sala n es el destino, no tiene llave ni puerta). Para la sala i:

Sala (i)Llave en s[...]Puerta en s[...]Fórmula
1s[0]s[1]2×(1−1)=0, +1=1
2s[2]s[3]2×(2−1)=2, +1=3
3s[4]s[5]2×(3−1)=4, +1=5
4s[6]s[7]2×(4−1)=6, +1=7
............
is[2*(i−1)]s[2*(i−1)+1]patrón general

Dicho de otra forma: cada sala ocupa 2 posiciones en la cadena. La sala 1 ocupa s[0] y s[1], la sala 2 ocupa s[2] y s[3], etc. Dentro de esas 2 posiciones, la primera siempre es la llave (minúscula) y la segunda siempre es la puerta (mayúscula).

La sala n es el destino final — no tiene llave ni puerta. Por eso la cadena tiene largo 2×(n−1) y no 2×n.

El problema concreto: llevar un inventario de llaves

Mientras recorrés las salas, necesitás llevar la cuenta de cuántas llaves tenés de cada tipo. Es como un inventario de un juego:

Tu inventario en un momento dado: llaves tipo 'a': 2 llaves tipo 'b': 0 llaves tipo 'f': 1 ... (26 tipos posibles, uno por cada letra)

¿Cómo representar eso en C++? Con un array de 26 enteros — uno por cada letra del abecedario:

int inventario[26] = {};  // todos arrancan en 0 (no tengo ninguna llave)

// inventario[0] = llaves tipo 'a'
// inventario[1] = llaves tipo 'b'
// inventario[2] = llaves tipo 'c'
// ...
// inventario[25] = llaves tipo 'z'

El problema: cuando leo un carácter de la cadena, por ejemplo 'f', necesito saber que corresponde a la posición 5 del array. No puedo hacer inventario['f'] porque 'f' vale 102 en ASCII y mi array solo tiene 26 posiciones.

La solución: restarle 'a'. Eso convierte cualquier minúscula en un número de 0 a 25:

Letra'a''b''c''d''e''f'...'z'
letra - 'a'012345...25

Ahora sí funciona: inventario['f' - 'a'] es inventario[5]. Perfecto.

Cuando recogés una llave (minúscula)

char llave = s[i];  // ej: 'f'

inventario[llave - 'a']++;   // 'f'-'a' = 5 → inventario[5]++
                               // Ahora tengo una llave 'f' más

Cuando enfrentás una puerta (mayúscula)

La puerta es mayúscula (ej: 'F'). Necesitás buscar en la misma posición del inventario que la llave 'f'. El truco: para mayúsculas, restás 'A' en vez de 'a':

Puerta'A''B''C''D''E''F'...'Z'
puerta - 'A'012345...25

'F' - 'A' = 5 y 'f' - 'a' = 5 — dan el mismo número. Así caen en la misma posición del inventario.

char puerta = s[i + 1];  // ej: 'F'

if (inventario[puerta - 'A'] > 0) {
    // 'F'-'A' = 5 → inventario[5] > 0? → tengo llave 'f'!
    inventario[puerta - 'A']--;  // uso una llave (se gasta)
} else {
    // inventario[5] == 0 → no tengo llave 'f'
    // necesito pedir una llave adicional
}
💡 Resumen: para qué sirve restar 'a' o 'A'

Es solo para convertir una letra en un número que sirva de índice en un array de 26 posiciones. Nada más. La razón por la que - 'a' y - 'A' dan el mismo resultado para la misma letra es que las minúsculas y mayúsculas están en el mismo orden en ASCII (a,b,c... y A,B,C...), solo que empiezan en números diferentes (97 vs 65). Al restar la base, esa diferencia desaparece.

⚠️ No podés usar tolower(), toupper() ni <cctype>

Esas funciones no están disponibles con las librerías permitidas (<iostream>, <string>, <cstring>). Restar 'a' o 'A' no necesita ninguna librería — funciona siempre.

¿Qué tipo de "tabla" necesitás para esto?

La letra dice que tenés que usar una tabla de hash o técnica de hashing. Pero antes de elegir hash abierto o cerrado, pensá en las características del problema:

  • ¿Cuántas claves posibles hay? Solo 26 (las letras a-z). No son miles ni millones — son exactamente 26.
  • ¿Sabés de antemano cuántas son? Sí, siempre son 26. No cambia con la entrada.
  • ¿Necesitás insertar/eliminar claves dinámicamente? No — las 26 letras existen siempre, solo cambia el contador asociado.
  • ¿Puede haber colisiones? Pensá: si sabés que las claves son exactamente las letras a-z y son solo 26... ¿se puede construir una función que mapee cada clave a una posición única sin colisiones?
💡 Pensá en esto

Cuando la cantidad de claves posibles es fija y chica (26 letras), y tenés una forma de convertir cada clave a un número único (ej: 'a'→0, 'b'→1, ..., 'z'→25), eso es una función de hash perfecta — sin colisiones, acceso O(1) garantizado en peor caso (no promedio). No necesitás manejar colisiones, ni elegir entre abierto o cerrado, porque colisiones no existen.

Comparalo con el ejercicio 2 donde las claves son nombres de personas (strings arbitrarios) — ahí sí necesitás una hash table real con manejo de colisiones.

⚠️ La complejidad dice O(N) peor caso, no promedio

Hash abierto y cerrado dan O(1) promedio pero O(N) peor caso (cuando hay muchas colisiones). Si la letra pide O(N) peor caso total, necesitás que cada operación sobre tu estructura sea O(1) peor caso, no promedio. Un array directo indexado por letra te da eso — hash abierto/cerrado no.

¿Cuál es la clave y cuál es el valor?

Cuando enfrentás una puerta 'F', la pregunta que hacés es: "¿cuántas llaves de tipo F tengo?"

  • Clave (key): la letra (a-z) — lo que buscás.
  • Valor (value): el contador — cuántas llaves de ese tipo tenés.

Es como una tabla de búsqueda: le das una letra, te devuelve un número.

¿Por qué esto es un hash y no "solo un array"?

Preguntate: ¿qué es una función de hash? Es una función que toma una clave y devuelve un índice (una posición en una tabla).

'f' - 'a' = 5

Eso ES una función de hash. Toma una clave ('f') y devuelve un índice (5). Y el array donde guardás los contadores ES la tabla.

Por eso la letra dice "tabla de hash o técnica de hashing" — un array indexado con letra - 'a' califica como técnica de hashing.

Paso 4 — ¿Por qué este ejercicio necesita una hash table real?

Ejercicio 1 vs Ejercicio 2

En el ejercicio 1, las claves eran letras a-z (26 posibles, fijas). Acá las claves son nombres de personas:

  • Puede haber colisiones — dos nombres diferentes podrían dar el mismo índice de hash.

Por eso acá sí necesitás una hash table con manejo de colisiones. No alcanza con un array simple.

¿Qué es la clave y qué es el valor?

Cuando procesás una ronda con "ana 100", la pregunta es: "¿cuál es el puntaje acumulado de ana?"

  • Clave: el nombre del participante (string).
  • Valor: su puntaje acumulado (int).

Operaciones que necesitás: buscar un nombre (¿ya existe?), insertar un nombre nuevo, actualizar su puntaje. Todo en O(1) promedio → hash table.

🚨 La letra lo dice explícitamente

"Utilizar una tabla de hash cerrada (direccionamiento abierto) con doble hashing"

No hay opción — tenés que usar hash cerrada (todo en un solo array, sin listas enlazadas) con doble hashing para resolver colisiones.

¿Qué es hash cerrada con doble hashing?

Hash cerrada (direccionamiento abierto): todos los elementos se guardan dentro del array. No hay listas enlazadas. Si la posición que te da la función de hash ya está ocupada (colisión), buscás otra posición libre dentro del mismo array.

Doble hashing: la forma de buscar la siguiente posición cuando hay colisión. Usás dos funciones de hash — h₁ te da la posición inicial, h₂ te da el "salto" entre intentos:

posición(k, i) = ( h₁(k) + i × h₂(k) ) % tamaño_tabla

Donde i es el número de intento (0, 1, 2, 3...). En el primer intento (i=0), la posición es h₁(k). Si hay colisión, probás h₁(k) + h₂(k). Si sigue ocupado, h₁(k) + 2×h₂(k). Y así.

¿Por qué doble hashing y no lineal o cuadrático?

  • Lineal (salto = 1, 2, 3...): genera "clustering" — las colisiones se agrupan y forman bloques que empeoran la performance.
  • Cuadrático (salto = 1, 4, 9...): reduce el clustering pero puede no visitar todas las posiciones.
  • Doble hashing (salto = h₂(k)): cada clave tiene su propio salto → la distribución es mucho más uniforme. Es la mejor de las tres.

Consideraciones para la implementación

  • h₂(k) nunca puede dar 0 — si el salto es 0, te quedás en la misma posición para siempre (loop infinito). Una forma de evitarlo: h₂(k) = primo - (h₁(k) % primo) donde primo es un primo menor al tamaño de la tabla.
  • El tamaño de la tabla debe ser primo — para que h₂ y el tamaño sean coprimos y se visiten todas las posiciones.
  • Cuántos participantes puede haber: hasta n (10⁶) — uno nuevo por ronda. Tu tabla necesita ser más grande que eso para mantener un factor de carga bajo.

Paso 5 — Las dos funciones de hash

Para doble hashing necesitás dos funciones de hash distintas, las dos para strings:

h₁ — la función de hash principal

Da la posición inicial en la tabla:

int h1(string nombre, int tamanioTabla) {
    int hash = 0;
    for (int i = 0; i < nombre.length(); i++) {
        hash = (hash * 31 + nombre[i]) % tamanioTabla;
    }
    return hash;
}

h₂ — la función de salto

Da cuánto saltás cuando hay colisión. Tiene que ser distinta a h₁ y nunca dar 0:

int h2(string nombre, int primo) {
    int hash = 0;
    for (int i = 0; i < nombre.length(); i++) {
        hash = (hash * 37 + nombre[i]) % primo;  // 37, no 31 — distinta a h1
    }
    return primo - hash;  // garantiza resultado entre 1 y primo (nunca 0)
}

primo es un número primo menor al tamaño de la tabla. El primo - hash garantiza que el resultado es siempre ≥ 1 (nunca 0).

¿Cómo se usan juntas?

Cuando buscás o insertás una clave en la tabla:

int pos = h1(nombre, tamanioTabla);
int salto = h2(nombre, primo);
int intento = 0;

while (tabla[pos] está ocupada Y tabla[pos].nombre != nombre) {
    intento++;
    pos = (h1(nombre, tamanioTabla) + intento * salto) % tamanioTabla;
}
// pos ahora es: la posición donde está el nombre, o una posición libre
  • ¿Por qué 31? Es un número primo que da buena distribución. Otros valores comunes: 37, 53, 97.
  • ¿Por qué el % tamanioTabla? Para que el resultado caiga dentro del rango del array (0 a tamanioTabla-1).
  • nombre[i] es un char, que en C++ es un número (su valor ASCII). Eso permite multiplicar y sumar.
⚠️ Recordá: solo <iostream>, <string>, <cstring>

No podés usar unordered_map, map, ni ninguna otra estructura de la STL. Tenés que implementar la hash table vos mismo — array, función de hash, manejo de colisiones, todo.

// ¿A tiene más prioridad que B?
// Devuelve true si A debería atenderse antes que B
bool tieneMasPrioridad(Paciente a, Paciente b) {
    if (a.urgencia != b.urgencia)
        return a.urgencia > b.urgencia;  // mayor urgencia primero
    if (a.hora != b.hora)
        return a.hora < b.hora;        // menor hora primero

Esta función va a ser la que tu heap use para decidir quién sube y quién baja — en vez de comparar arr[pos] < arr[posPadre] con ints, comparás con tieneMasPrioridad(arr[pos], arr[posPadre]).

7 Ejercicio 5 — Formación de equipos

PENDIENTE — usa DFS/BFS sobre un grafo no dirigido
📐 Problema en una oración

V estudiantes con A incompatibilidades. ¿Se pueden dividir en 2 equipos sin que dos incompatibles queden juntos?

⚠️ Restricciones de la letra
  • Complejidad: O(V+A) peor caso.
  • Límites: 1 ≤ V ≤ 10⁵, 0 ≤ A ≤ 10⁵.
  • El grafo puede no ser conexo.
  • Salida: SI o NO.

Paso 1 — Entender la entrada

Formato

4 4              // V=4 estudiantes, A=4 incompatibilidades
1 2              // 1 y 2 no pueden estar juntos
2 3
3 4
4 1
int v, a;
cin >> v >> a;
// Construir grafo no dirigido con v vértices
for (int i = 0; i < a; i++) {
    int u, w;
    cin >> u >> w;
    // agregar arista u-w (bidireccional)
}

Los vértices son números (1 a V) → podés usar directamente la ListaAdyacencia del profesor. No necesitás hash table.

Paso 2 — ¿Qué significa "dividir en 2 equipos"?

Pensalo con colores

Dividir en 2 equipos es lo mismo que pintar cada nodo con uno de 2 colores (ej: rojo y azul) de forma que ninguna arista conecte dos nodos del mismo color.

Ejemplo: 1—2, 2—3, 3—4, 4—1 Pintamos: 1=rojo, 2=azul, 3=rojo, 4=azul Arista 1—2: rojo—azul ✓ Arista 2—3: azul—rojo ✓ Arista 3—4: rojo—azul ✓ Arista 4—1: azul—rojo ✓ → SE PUEDE (SI) Ejemplo: 1—2, 2—3, 3—1 (triángulo) Pintamos: 1=rojo, 2=azul, 3=??? 3 está conectado a 2 (azul) → 3 tiene que ser rojo 3 está conectado a 1 (rojo) → 3 tiene que ser azul Contradicción! → NO SE PUEDE (NO)

Paso 3 — ¿Cómo resolverlo?

DFS con 2 colores (el tip del profesor: usar DFS, no BFS)

La idea: hacer DFS, y al visitar cada nodo pintarlo del color opuesto al de su padre.

  • Empezás con un nodo, lo pintás de color 0.
  • Para cada vecino: si no está pintado, lo pintás con el color opuesto y seguís con DFS.
  • Si un vecino ya está pintado con el mismo color que el nodo actual → es imposible → NO.
  • Si terminás de recorrer todo sin conflictos → SI.

¿Qué usás en vez de bool* visitado?

En vez de un array de bools, usás un array de ints:

int* color = new int[v + 1];
for (int i = 0; i <= v; i++) color[i] = -1;  // -1 = no pintado

// color[i] == -1 → no visitado (equivale a visitado[i] == false)
// color[i] == 0  → equipo A
// color[i] == 1  → equipo B

El color opuesto de c es 1 - c (si es 0 → 1, si es 1 → 0).

🚨 El grafo puede no ser conexo

Puede haber múltiples componentes conexas. Tenés que hacer DFS desde cada nodo no visitado — igual que en cantComponentes. Si ALGUNA componente falla, la respuesta es NO.

💡 Lo que ya tenés implementado te sirve
  • Grafo: ListaAdyacencia del profesor (los vértices ya son números).
  • DFS: el mismo DFS_rec, pero en vez de visitado[origen] = true, hacés color[origen] = c.
  • Componentes: el loop de cantComponentes para manejar grafos no conexos.
Ejercicio 5 — Resumen
Entrada: V y A, luego A pares (u, v). Salida: SI o NO. Idea: DFS con 2 colores. Pintar cada vecino con el color opuesto. Si un vecino ya tiene el mismo color → NO. Array: int* color con -1 (no visitado), 0 (equipo A), 1 (equipo B). Color opuesto: 1 - c.
Grafo no conexo: hacer DFS desde cada nodo con color == -1 (como cantComponentes). Complejidad: O(V+A) — un DFS que toca cada nodo y arista una vez.
